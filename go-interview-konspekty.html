Алгоритмы и сложность

Big-O — оценка сложности

TL;DR: Big-O показывает, как растёт время (или память) алгоритма при увеличении размера входных данных n, игнорируя константы.

Нас интересует не «сколько секунд», а «во сколько раз станет дольше, если данных станет в 10 раз больше». Константы и мелочи отбрасываем — важна форма роста.

ОбозначениеНазваниеПример
O(1)константноедоступ к элементу массива по индексу, get из хеш-таблицы
O(log n)логарифмическоебинарный поиск, поиск в сбалансированном дереве
O(n)линейноеперебор всего массива
O(n log n)«почти линейное»хорошие сортировки (quicksort, merge)

O(1) — достать книгу с полки по номеру; O(n) — пролистать всю стопку; O(log n) — искать слово в словаре, деля его пополам; O(n²) — сравнить каждого с каждым в комнате.

Ещё говорят: «сложность в худшем случае» (worst case), «в среднем» (average), «амортизированная» — усреднённая по многим операциям (например, append в слайс: обычно O(1), изредка O(n) при реаллокации, но амортизированно O(1)).

Быстрая сортировка (quicksort)

Короткий ответ: берём «опорный» элемент (pivot), раскидываем меньшие влево, бо́льшие вправо (partition), и рекурсивно сортируем половинки. В среднем O(n·log n), на практике очень быстрая; в худшем случае — O(n²). Сортирует «на месте», неустойчивая.

Раскладываем колоду по росту: берём карту-эталон, все ниже — в левую кучу, выше — в правую, потом так же сортируем каждую кучу.

Как это работает — механизм, пример и когда O(n²)

Механизм: partition (разбиение)

Сердце алгоритма — partition: за один проход переставить элементы так, чтобы слева от pivot оказались меньшие, справа — бо́льшие. Прогон (pivot = последний, схема Ломуто), i — граница «меньших»:

[7, 2, 5, 1, 3]   pivot = 3
 j=0: 7>3  — пропускаем
 j=1: 2<3  → [2, 7, 5, 1, 3]  (i=0)
 j=2: 5>3  — пропускаем
 j=3: 1<3  → [2, 1, 5, 7, 3]  (i=1)
 ставим pivot на место i+1:  [2, 1, 3, 7, 5]
                                     ^ pivot
 слева меньшие [2,1] · справа бо́льшие [7,5] → рекурсия по ним

Почему в среднем O(n·log n)

Если pivot делит массив примерно пополам, глубина рекурсии ~log n, а на каждом уровне суммарно проходим все n элементов:

Когда вырождается в O(n²)

Если pivot почти всегда оказывается минимумом или максимумом, деления «пополам» нет — одна часть пустая, вторая размера n−1. Классика: уже отсортированный вход и pivot = последний (или первый) элемент.

Вход: [1, 2, 3, 4, 5], pivot = последний
[1,2,3,4,5] → pivot 5 в конце, слева [1,2,3,4], справа []
[1,2,3,4]   → pivot 4,             слева [1,2,3],   справа []
[1,2,3]     → pivot 3,             слева [1,2],     справа []
...
каждый шаг отрезает всего 1 элемент → n уровней → O(n²)

Как защищаются: выбирать pivot случайно или как медиану трёх (первый/средний/последний); при слишком глубокой рекурсии — переключаться на heapsort. Именно это делает pdqsort — сортировка по умолчанию в Go.

Код (Ломуто)

func quicksort(a []int) {
    if len(a) < 2 {
        return
    }
    p := partition(a)
    quicksort(a[:p])
    quicksort(a[p+1:])
}

func partition(a []int) int {
    pivot := a[len(a)-1] // опорный — последний
    i := 0
    for j := 0; j < len(a)-1; j++ {
        if a[j] < pivot {
            a[i], a[j] = a[j], a[i]
            i++
        }
    }
    a[i], a[len(a)-1] = a[len(a)-1], a[i] // pivot на своё место
    return i
}

Кучная сортировка (heapsort)

Короткий ответ: строим из массива кучу (в корне всегда максимум) и по очереди вынимаем максимумы в конец. Всегда O(n·log n), без «плохих» случаев, «на месте» (память O(1)); неустойчивая.

Турнир на выбывание: наверх всплывает сильнейший, забираем его, остальные снова разыгрывают первое место.

Механизм, схема и код

Как это работает

Куча (binary heap) — это массив, который читается как двоичное дерево: у узла с индексом i дети — 2i+1 и 2i+2. Свойство max-heap: родитель ≥ детей, поэтому в корне (индекс 0) — максимум.

Алгоритм: (1) построить max-heap из массива (просеять узлы снизу вверх); (2) поменять корень (максимум) с последним элементом, уменьшить кучу на 1 и «просеять» новый корень вниз; (3) повторять, пока куча не опустеет. Максимумы копятся в конце → массив отсортирован.

Код (siftDown)

func heapSort(a []int) {
    n := len(a)
    for i := n/2 - 1; i >= 0; i-- { // построить кучу
        siftDown(a, i, n)
    }
    for end := n - 1; end > 0; end-- { // вынимать максимум в конец
        a[0], a[end] = a[end], a[0]
        siftDown(a, 0, end)
    }
}

func siftDown(a []int, i, n int) {
    for {
        l, r, big := 2*i+1, 2*i+2, i
        if l < n && a[l] > a[big] { big = l }
        if r < n && a[r] > a[big] { big = r }
        if big == i { return }
        a[i], a[big] = a[big], a[i]
        i = big
    }
}

Сортировка вставками (insertion sort)

Короткий ответ: берём элементы по одному и вставляем каждый на место среди уже отсортированных слева. O(n²) в худшем, но очень быстрая на маленьких и почти отсортированных массивах; устойчивая, на месте.

Так сортируют карты в руке: берёшь новую карту и вставляешь её между уже разложенными по порядку.

Прогон и код

Прогон: [5, 2, 4, 1]

[5 | 2, 4, 1]   старт: слева «отсортировано» только 5
вставляем 2  → [2, 5 | 4, 1]
вставляем 4  → [2, 4, 5 | 1]
вставляем 1  → [1, 2, 4, 5]
(на каждом шаге сдвигаем бо́льшие вправо и кладём элемент на освободившееся место)

Код

func insertionSort(a []int) {
    for i := 1; i < len(a); i++ {
        key := a[i]
        j := i - 1
        for j >= 0 && a[j] > key { // сдвигаем бо́льшие вправо
            a[j+1] = a[j]
            j--
        }
        a[j+1] = key // вставляем на место
    }
}

Почему её любят в гибридах: на почти отсортированном входе работает почти за O(n) (внутренний цикл почти не крутится), и на коротких кусках обгоняет «умные» сортировки за счёт простоты и отсутствия накладных расходов. Поэтому её вызывают на мелких подмассивах внутри pdqsort.

Спрашивают на собесе: «лучший и худший случай?» (O(n) на почти отсортированном / O(n²) на обратно отсортированном); «устойчива ли?» (да); «зачем она в быстрых сортировках?» (мелкие куски).

Сортировка слиянием (merge sort)

Короткий ответ: делим массив пополам, рекурсивно сортируем половины, потом сливаем два отсортированных куска в один. Всегда O(n·log n), устойчивая, но нужна доп. память O(n).

Две отсортированные стопки карт складываем в одну: сравниваем верхние карты и берём меньшую — и так пока стопки не кончатся.

Механизм, схема и код

Как это работает

Два шага: divide — режем пополам до кусков длины 1 (они уже «отсортированы»); merge — на обратном ходе сливаем пары отсортированных кусков. Слияние идёт одновременно по двум спискам: на каждом шаге берём меньшую из «голов».

Про устойчивость и память

Устойчивая (stable): при слиянии берём из левого куска при равенстве (l[i] <= r[j]) — значит равные элементы сохраняют исходный порядок (два «Иванова» не поменяются местами). В Go sort.Stable / slices.SortStableFunc используют вариант merge sort. Минус — буфер O(n) под слияние.

Код

func mergeSort(a []int) []int {
    if len(a) < 2 {
        return a
    }
    mid := len(a) / 2
    left := mergeSort(a[:mid])
    right := mergeSort(a[mid:])
    return merge(left, right)
}

func merge(l, r []int) []int {
    res := make([]int, 0, len(l)+len(r))
    i, j := 0, 0
    for i < len(l) && j < len(r) {
        if l[i] <= r[j] { // <= → устойчивость
            res = append(res, l[i]); i++
        } else {
            res = append(res, r[j]); j++
        }
    }
    res = append(res, l[i:]...)
    res = append(res, r[j:]...)
    return res
}

pdqsort (сортировка по умолчанию в Go)

Короткий ответ: «pattern-defeating quicksort» — гибрид: основа quicksort + insertion sort на мелких кусках + heapsort как страховка от O(n²). Это дефолт sort.Sort, slices.Sort (неустойчивая).
Как комбинирует три сортировки и «побеждает» плохие данные
  • Основа — quicksort: быстрый в среднем, сортирует на месте.
  • Мелкие куски → insertion sort: на подмассивах из нескольких элементов он быстрее.
  • Слишком глубокая рекурсия → heapsort: если quicksort начал вырождаться (риск O(n²)), pdqsort переключается на heapsort, который гарантирует O(n·log n).

Почему «pattern-defeating»: обычный quicksort можно «сломать» специально подобранным или уже отсортированным входом (см. пример O(n²) в карточке quicksort). pdqsort это детектирует: выбирает pivot по медиане, тасует элементы при подозрительных паттернах и ограничивает глубину рекурсии — поэтому сложность не вырастает до квадратичной ни на каком входе.

Для устойчивой сортировки Go применяет block merge sort (sort.Stable, slices.SortStableFunc).

Бинарный (двоичный) поиск

Короткий ответ: поиск в отсортированном массиве делением пополам: сравниваем с серединой и каждый раз отбрасываем половину. O(log n).

Угадать число от 1 до 1000 за ~10 вопросов «больше/меньше»: каждый ответ убирает половину вариантов.

Код, прогон и подводные камни

Код (итеративно)

func binSearch(a []int, target int) int {
    lo, hi := 0, len(a)-1
    for lo <= hi {
        mid := lo + (hi-lo)/2 // так не переполнится (не (lo+hi)/2)
        switch {
        case a[mid] == target:
            return mid
        case a[mid] < target:
            lo = mid + 1 // искомое правее
        default:
            hi = mid - 1 // искомое левее
        }
    }
    return -1 // не найдено
}

Прогон: ищем 7 в [1, 3, 5, 7, 9, 11]

lo=0 hi=5  mid=2  a[2]=5  <7 → lo=3
lo=3 hi=5  mid=4  a[4]=9  >7 → hi=3
lo=3 hi=3  mid=3  a[3]=7  =7 → нашли, индекс 3

Тонкости

  • Только на отсортированных данных — иначе результат бессмыслен.
  • Переполнение: mid := (lo+hi)/2 может переполнить int на огромных индексах — классический баг; пишем lo + (hi-lo)/2.
  • Инвариант границ: искомое всегда в [lo..hi]; выход из цикла при lo > hi — не нашли.
  • В Go готовое: slices.BinarySearch(s, x) (вернёт индекс и found bool); sort.Search(n, f) — обобщённый поиск «первого, где условие истинно» (удобно для нижней границы / вставки).
Спрашивают на собесе: «сложность и почему log n?»; «как избежать переполнения при вычислении середины?»; «как найти первое вхождение / позицию для вставки?» (нижняя граница, sort.Search); «что если массив не отсортирован?».

Массив и связный список

Короткий ответ: массив — элементы лежат подряд в памяти: доступ по индексу O(1) и отличная кэш-локальность, но вставка/удаление в середину — O(n) (сдвиг) и размер фиксирован. Связный список — элементы разбросаны и связаны ссылками: вставка/удаление узла — O(1), но только если этот узел уже на руках (найти позицию — всё равно O(n)); доступ по индексу — O(n), кэш плохой.

Массив — ряд пронумерованных ящиков вплотную (сразу прыгаешь к нужному номеру). Список — записки, где каждая говорит «следующая лежит вон там».

Хеш-таблица (hash table / hashmap)

Короткий ответ: структура «ключ → значение» с доступом за O(1) в среднем: ключ прогоняют через хеш-функцию, число приводят к размеру массива → индекс «корзины» (bucket). Худший случай — O(n) при массовых коллизиях.

Гардероб: по номеру билета (хеш) сразу знаешь нужный крючок, не перебирая все пальто.

Механизм, коллизии, load factor и map в Go

Как кладём и достаём

set(key, value): (1) хеш-функция превращает ключ в большое число; (2) число приводится к размеру массива (обычно остаток от деления) → индекс корзины; (3) кладём пару туда. get(key) — тот же путь до корзины, затем сравнение ключей внутри неё.

Коллизии

Два разных ключа могут дать один индекс. Способы разрулить:

  • Цепочки (chaining): в корзине — список всех столкнувшихся пар (так в Go: overflow-бакеты).
  • Открытая адресация (open addressing): занято — ищем следующую свободную ячейку (линейное/квадратичное пробирование); все пары лежат в самом массиве.

Load factor и рехешинг

Load factor = число элементов / число корзин — насколько таблица «набита». Чем выше, тем длиннее цепочки и медленнее операции. Порог превышен → рехешинг: создаём массив побольше и перекладываем всё (пересчитывая индексы). Ориентиры: для открытой адресации держат load factor < ~0.7, для цепочек допускают ~1 и выше.

Эвакуация (инкрементальный рехешинг в Go)

Когда map решает вырасти, она не перекладывает всё сразу (это была бы длинная пауза). Заводит новый массив бакетов вдвое больше, а старый (oldbuckets) держит рядом. На каждой записи/удалении переносит по чуть-чуть — пару старых бакетов в новый; это и есть «эвакуация» (growWork). Пока идёт рост, чтение смотрит и в старый, и в новый массив. Два вида роста: удвоение (реально много элементов, load factor превышен) и рост того же размера (накопилось много overflow-бакетов от удалений → «уплотнить»).

Load factor в Go

Порог роста — load factor ~6.5 (в среднем 6.5 элементов на бакет из 8 слотов). Выше — цепочки длиннее, операции медленнее, поэтому запускается эвакуация в массив побольше.

Что может быть ключом

Ключ должен быть comparable (сравним через ==):

  • Можно: числа, строки, bool, указатели, каналы, интерфейсы, а также структуры и массивы из сравнимых полей.
  • Нельзя (ошибка компиляции): слайс, map, функция — они не сравнимы.
  • Тонкости: ключ типа interface{} компилируется, но если положить внутрь несравнимое (слайс) — паника в рантайме; NaN как ключ «не равен сам себе» → такой элемент потом не достать.

Порядок обхода и сортировка

Порядок итерации по map намеренно случайный (Go рандомизирует, чтобы никто не полагался на него). Чтобы обойти в отсортированном виде — собрать ключи в слайс, отсортировать, идти по нему:

keys := make([]string, 0, len(m))
for k := range m {
    keys = append(keys, k)
}
sort.Strings(keys)
for _, k := range keys {
    fmt.Println(k, m[k])
}

Ещё особенности map в Go

  • Внутри — цепочки: бакет на 8 слотов + tophash (старшие биты хеша для быстрого сравнения) + overflow-бакеты; с Go 1.24 — Swiss Tables.
  • nil-map: читать можно (вернёт нулевое значение), писать нельзя (panic: assignment to entry in nil map) — нужен make.
  • Comma-ok: v, ok := m[k] — отличить «ключа нет» от «значение = нулевое».
  • Значение map не адресуемо: &m[k] нельзя; m[k].field = x для структуры-значения — ошибка (класть указатель или перезаписывать целиком).
  • map передаётся дёшево — это указатель на hmap; изменения видны всем держателям.
  • Не потокобезопасна: конкурентная запись → fatal error: concurrent map writes (не паника, recover не спасёт). Лечится мьютексом или sync.Map (для read-mostly).
Спрашивают на собесе: «средняя и худшая сложность и почему?»; «что такое load factor и когда map растёт?» (~6.5); «как устроена эвакуация?» (инкрементально, oldbuckets→newbuckets по кусочку); «что может быть ключом, а что нет?» (comparable; не слайс/map/функция); «какой порядок обхода и как отсортировать?»; «почему падает при конкурентной записи и как чинить?».

Хеш-функция

Короткий ответ: превращает данные любого размера в число/строку фиксированной длины. Одинаковый вход → одинаковый выход; хорошая хеш-функция распределяет значения равномерно и быстра.
Свойства, примеры и криптостойкость

Свойства хорошей хеш-функции

  • Детерминированность — тот же вход даёт тот же хеш.
  • Равномерность — значения раскиданы ровно, мало коллизий (лавинный эффект: смена одного бита входа сильно меняет хеш).
  • Скорость — для хеш-таблиц считается на каждый доступ.

Коллизия неизбежна

Входов бесконечно много, а хешей — конечное число, поэтому разные входы иногда дают один хеш. Задача не «убрать коллизии» (невозможно), а сделать их редкими и равномерными.

Какие бывают

  • Быстрые (не крипто): FNV, crc32, xxhash — для хеш-таблиц и контрольных сумм.
  • Криптографические: sha256 — целостность, подписи; по хешу нельзя восстановить вход и трудно подобрать коллизию.
  • Сломанные: md5, sha1 — годятся только как быстрые контрольные суммы, не для безопасности.
  • Для паролей: не обычный хеш, а медленные bcrypt / argon2 + соль (случайная добавка против радужных таблиц).
Спрашивают на собесе: «какие свойства у хорошей хеш-функции?»; «почему коллизии неизбежны?»; «чем крипто-хеш отличается от обычного?»; «чем и как хешируют пароли и зачем соль?»; «можно ли md5 для паролей?» (нет).

Деревья и B-tree

Короткий ответ: дерево — иерархия «родитель → дети». Сбалансированное дерево поиска даёт O(log n) на поиск/вставку. B-tree/B+-tree — «широкие» деревья, оптимизированные под диск; на них построены индексы БД.

B-tree — оглавление толстой книги с разделами и подразделами: пара переходов — и ты на нужной странице.

BST, балансировка, B-tree vs B+-tree — со схемой

Бинарное дерево поиска (BST)

У каждого узла ≤ 2 детей: слева меньшие значения, справа бо́льшие. Поиск идёт вниз, на каждом шаге отбрасывая половину → O(log n) — если дерево сбалансировано. Если вставлять уже отсортированные данные в наивный BST, он вырождается в список (O(n)). Балансировку держат AVL и red-black деревья (при вставке «поворачивают» узлы).

B-tree / B+-tree — под диск

У узла много детей (сотни). Дерево «низкое и широкое» — мало уровней, значит мало обращений к диску: одно чтение узла = одна страница диска.

Разница B-tree vs B+-tree: в B-tree данные лежат в любом узле; в B+-tree — только в листьях, а внутренние узлы хранят лишь ключи-разделители, и листья связаны в список → удобны range-запросы и упорядоченный обход. Именно B+-tree — стандарт для индексов БД (Postgres, MySQL): за 3–4 чтения находишь запись среди миллионов.

Спрашивают на собесе: «сложность поиска в BST и когда она портится?» (вырождение в список); «как балансируют?» (AVL/red-black); «почему в БД именно B-tree, а не BST?» (низкое дерево = мало чтений с диска, узел = страница); «B-tree vs B+-tree?» (данные в листьях + связанные листья → range-запросы); «B-tree vs хеш-индекс?» (хеш — только точное равенство, B-tree — ещё диапазоны и сортировка).

Куча (binary heap) — структура данных

Короткий ответ: дерево, где родитель всегда ≥ (max-heap) или ≤ (min-heap) детей. Минимум/максимум — в корне: посмотреть O(1), извлечь/добавить O(log n). Не путать с «кучей» как областью памяти.

Пирамида, где начальник всегда «главнее» подчинённых; на вершине — самый главный.

Устройство, операции, применения — со схемой

Как устроена

Хранится в обычном массиве и читается как двоичное дерево: у узла с индексом i дети — 2i+1 и 2i+2, родитель — (i-1)/2. Дерево всегда «полное» (заполняется по уровням), поэтому высота ~log n.

Операции

  • push (вставка): кладём элемент в конец массива и «всплываем» вверх, меняясь с родителем, пока порядок не восстановится — O(log n).
  • pop (извлечь корень): берём корень, ставим на его место последний элемент и «просеиваем» вниз, меняясь с меньшим/большим ребёнком — O(log n).
  • peek (посмотреть min/max): это корень — O(1).

Где применяют

Очередь с приоритетом (priority queue), heapsort, поиск k наибольших/наименьших, планировщики задач и таймеры, алгоритм Дейкстры (кратчайшие пути). В Go — пакет container/heap (реализуешь интерфейс Len/Less/Swap/Push/Pop).

Спрашивают на собесе: «чем куча-структура отличается от кучи-памяти?»; «как массив представляет кучу (формулы индексов)?»; «сложность вставки/извлечения/просмотра?»; «как найти k наибольших за O(n·log k)?» (min-heap размера k); «где куча в реальных задачах?» (priority queue, Дейкстра, таймеры).

LRU-кэш

Короткий ответ: кэш фиксированного размера, который при переполнении выбрасывает давно не используемый элемент (Least Recently Used). Все операции — O(1).

Стопка бумаг на столе: взял лист — кладёшь наверх; места нет — выбрасываешь самый нижний (давно не трогали).

Как устроен (частая задача на собесе) — со схемой

Хеш-таблица + двусвязный список

Комбинируют две структуры, потому что по отдельности каждая чего-то не умеет за O(1):

  • Хеш-таблица ключ → узел: мгновенный доступ к узлу за O(1).
  • Двусвязный список: хранит элементы по «свежести» (хвост — недавно использованные, голова — старые) и позволяет вырвать/переставить узел за O(1) — для этого и нужны обе ссылки.

Операции

  • get(key): нашли узел через map → переносим его в хвост (стал «свежим») → вернули значение.
  • put(key, val): есть — обновили и в хвост; нет — создали узел в хвосте и записали в map; если превысили размер — удалили голову (самый старый) и стёрли её ключ из map.

В Go: в stdlib готового LRU нет — берут hashicorp/golang-lru либо пишут сами на map + container/list. Это классическая задача на собесе — «реализуй LRU с O(1)».

Спрашивают на собесе: «реализуй LRU за O(1)» (map + двусвязный список); «почему именно двусвязный список?» (удалить узел из середины за O(1) нужны обе ссылки); «зачем тут ещё и map?» (найти узел за O(1)); «что вытесняем при переполнении?» (голову — LRU); «чем LRU отличается от LFU?» (LFU — по частоте, а не по свежести).

Биты, байты, ASCII

TL;DR: компьютер хранит всё числами. Бит — 0 или 1; байт — 8 бит (256 значений). Кодировка — договор, какое число обозначает какой символ. ASCII — старейший такой договор.

ASCII (1963) кодирует 128 символов (латиница, цифры, знаки) числами 0–127, помещаясь в 1 байт. Например, 'A' = 65, 'a' = 97. Проблема: нет места для кириллицы, иероглифов, эмодзи.

Unicode и code point

TL;DR: Unicode — единый каталог всех символов мира (буквы, иероглифы, эмодзи). Каждому символу присвоен номер — code point (например, U+0041 = 'A', U+1F600 = 😀).

Unicode — это таблица соответствий «символ ↔ номер», а не способ хранения в байтах. Как именно записать эти номера в память — решает кодировка (UTF-8, UTF-16, UTF-32). В Go тип rune = один code point (int32).

UTF-8 — как символы превращаются в байты

TL;DR: UTF-8 — самый популярный способ записать Unicode в байты. Гениальность в том, что частые символы (латиница) занимают 1 байт, редкие — 2–4 байта, и старый ASCII-текст остаётся валидным UTF-8.

Переменная длина: символ занимает от 1 до 4 байт:

  • латиница/цифры (ASCII) — 1 байт (полная совместимость с ASCII!);
  • кириллица, большинство европейских — 2 байта;
  • иероглифы — 3 байта;
  • эмодзи, редкие знаки — 4 байта.

По первым битам байта понятно, сколько байт занимает символ и где он начинается — поэтому по UTF-8 можно идти вперёд/назад без таблиц.

Почему это важно для Go: строка — это байты в UTF-8. Поэтому len("привет") = 12 (по 2 байта на букву), а символов 6. Чтобы считать символы — utf8.RuneCountInString или пройтись range (он декодирует руны). Авторы Go Роб Пайк и Кен Томпсон — соавторы самого UTF-8, отсюда родная поддержка.

Как азбука Морзе: частые буквы — короткие сигналы, редкие — длинные; поток читается однозначно.

В банке: строки, руны, байты, итерация по строке.

Память и ОС

Стек и куча (области памяти)

Короткий ответ: стек — быстрая память под локальные переменные функции: кладётся при входе, выбрасывается при выходе (без уборки). Куча — общее хранилище для данных, которые должны жить дольше вызова; за ними убирает сборщик мусора (GC), поэтому она медленнее. Куда попадёт конкретная переменная, в Go решает компилятор через escape analysis: не «убегает» из функции → стек, «убегает» → куча.

Стек — стопка тарелок (берём/кладём только сверху, всё убирается разом). Куча — большой склад, где вещи лежат где попало и кто-то (GC) периодически вывозит мусор.

Подробный конспект — с примерами и схемами

Стек — «стопка тарелок»

На каждый вызов функции сверху кладётся её фрейм (аргументы, локальные переменные, адрес возврата). Вышли из функции — фрейм снимается. Работает по принципу LIFO (последним пришёл — первым ушёл): выделить память — просто сдвинуть указатель (O(1)), освободить — сдвинуть обратно, и вся память функции уходит разом, без всякого GC. Свой стек у каждой горутины.

Стек горутины умеет расти

Стек горутины стартует всего с ~2 КБ — поэтому в Go можно держать миллионы горутин (10 000 ≈ 20 МБ на старте), тогда как у потока ОС стек 1–8 МБ и их хватает лишь на тысячи. Если стека не хватило, рантайм выделяет вдвое больший блок, копирует туда старый и чинит все указатели (это состояние _Gcopystack); при желании стек и сжимается.

Историческая справка: до Go 1.3 стек был сегментным (цепочка 4-КБ кусочков), но от этого ушли из-за «hot split» (дёрганья рост/сжатие на границе) и плохой локальности кэша — теперь непрерывный стек с ростом копированием. Потолок ~1 ГБ (64-бит) / 250 МБ (32-бит); упёрлись → fatal error: stack overflow, обычно из-за бесконечной рекурсии.

Куча — «склад» под управлением рантайма

Куда сложнее: организована уровнями — arena (крупные блоки, взятые у ОС) → page (по 8 КБ) → span (участки под объекты одного размерного класса). Мелкие объекты (≤ 32 КБ) нарезаются из span-ов, крупные идут отдельными блоками. Когда нужна память: сначала ищем свободный кусок в free list нужного размера; нет — берём новый span; совсем нет — просим у ОС (mmap). А убирает ненужное GC — отсюда и накладные расходы.

Где живёт переменная? Escape analysis

Компилятор смотрит, «убегает» ли переменная за пределы функции. Не убегает → кладёт в стек (дёшево и безопасно); убегает → вынужден положить в кучу. Компилятор предпочитает стек и уводит в кучу только когда обязан. Посмотреть его решения: go build -gcflags='-m' main.go (-m=2 — подробнее).

func stackOK() int {
    x := 42       // не убегает → стек
    return x
}

func escapes() *int {
    x := 42       // &x переживёт функцию → куча
    return &x
}
// go build -gcflags=-m → во второй функции: "moved to heap: x"

Что чаще всего уводит в кучу:

  • Возврат указателя на локальную переменную (return &x) — она должна пережить функцию.
  • Упаковка в интерфейс — например fmt.Println(x) (параметр any) часто вызывает escape.
  • Захват замыканием, которое живёт дольше функции.
  • Передача указателя туда, где его сохранят (в структуру, канал, глобальную переменную).
  • Объект неизвестного/слишком большого размера (например слайс, растущий в рантайме).

Как беречь GC (важно под нагрузкой)

Меньше аллокаций в куче = быстрее код и меньше пауз GC: держать данные на стеке (не плодить лишние указатели), переиспользовать объекты (sync.Pool), меньше временных объектов в горячих циклах, заранее задавать ёмкость слайсов и мап (make([]T, 0, n)).

Виртуальная память и страницы

TL;DR: каждая программа думает, что владеет всей памятью — это виртуальная память. ОС незаметно переводит её адреса в реальные (физические), разбивая память на страницы (обычно 4 КБ).

Зачем: (1) изоляция — процессы не видят память друг друга (безопасность); (2) можно дать больше памяти, чем есть физически (лишнее — на диск, «своп»); (3) защита участков (только чтение / нельзя исполнять).

Страница (page) — минимальный блок для этого перевода. Таблица страниц хранит соответствие «виртуальная страница → физическая». Своп — выгрузка редких страниц на диск, когда RAM не хватает.

Виртуальные адреса — как номера в гостинице; ресепшн (ОС) знает, за какой реальной дверью какой гость, и может переселять их незаметно.

TLB и hugepages

TL;DR: TLB — маленький сверхбыстрый кэш процессора, который помнит недавние переводы «виртуальный адрес → физический», чтобы не лазить в таблицу страниц каждый раз.

Каждое обращение к памяти требует перевода адреса (см. виртуальная память). Если бы это был каждый раз поиск в таблице — было бы медленно. TLB (Translation Lookaside Buffer) кэширует последние переводы; при работе с близкими адресами почти всегда «попадание» → быстро.

Hugepages — большие страницы (2 МБ вместо 4 КБ). Меньше страниц → меньше записей → меньше промахов TLB. Полезно тяжёлым программам (базы данных), которые работают с гигабайтами.

Кэш процессора и локальность

TL;DR: процессор намного быстрее RAM, поэтому держит рядом маленький сверхбыстрый кэш (L1/L2/L3). Данные, лежащие рядом в памяти, читаются пачками — это «кэш-локальность».

Обращение к RAM в разы медленнее, чем к кэшу. Процессор подгружает данные «линиями» (обычно 64 байта): взял один элемент — соседние уже в кэше. Поэтому массив (элементы подряд) перебирается быстрее связного списка (элементы разбросаны), даже если по Big-O они равны.

Держать нужные книги на столе (кэш), а не бегать в дальний архив (RAM) за каждой.

Почему []T обычно быстрее []*T, а линейный поиск по массиву — на удивление шустрый.

Сборка мусора (GC): mark-sweep и трёхцветная пометка

Короткий ответ: сборщик мусора сам освобождает память, до которой из программы больше нельзя дотянуться. Go использует конкурентный (работает вместе с приложением) mark-and-sweep с трёхцветной пометкой; паузы stop-the-world — доли миллисекунды. GC в Go не поколенческий.

Уборка на вечеринке, не выгоняя гостей: помечаем, кто ещё нужен (за кого держатся), остальное убираем; охранник на входе (write barrier) следит, чтобы гости не прятали друг друга.

Подробный конспект — с примерами и схемами

Mark & Sweep — в двух словах

Mark (пометка): начиная с «корней» (локальные переменные горутин, глобальные переменные), GC обходит все ссылки и помечает всё достижимое как живое. Sweep (очистка): всё, что осталось непомеченным, — мусор, его память возвращается в кучу.

Трёхцветная пометка

Способ вести пометку, не останавливая программу надолго. Каждый объект красится в один из трёх цветов:

  • белый — ещё не посещён (кандидат в мусор);
  • серый — найден живым, но его ссылки ещё не просмотрены (это «фронт» обхода);
  • чёрный — и сам, и все его ссылки просмотрены (точно живой).

Сначала все белые, корни красим в серый. Дальше: берём серый объект → красим его ссылки в серый, а его самого — в чёрный. Повторяем, пока серые не кончатся. Оставшиеся белые недостижимы → их сметаем.

Почему нужен write barrier

GC работает одновременно с приложением, поэтому программа может «на ходу» переставить указатель и спрятать живой объект от сборщика (записать ссылку из уже чёрного в белый, который сборщик считает мусором). Write barrier — крохотный перехватчик записи указателей: он подкрашивает такой объект в серый, поддерживая инвариант «чёрный не ссылается на белый напрямую». Go использует гибридный write barrier (с Go 1.8) — за счёт него паузы стали суб-миллисекундными.

Фазы и stop-the-world

Почти вся работа идёт вместе с приложением. Полностью замереть (STW) нужно лишь на короткие моменты: включить write barrier в начале и согласовать результат в конце пометки (mark termination). Очистка (sweep) тоже конкурентная/ленивая.

Чем настраивается

  • GOGC (по умолчанию 100): GC запускается, когда куча выросла на GOGC% с прошлой сборки (т.е. примерно удвоилась). Больше значение → реже сборки, но больше памяти; GOGC=off выключает GC.
  • GOMEMLIMIT (с Go 1.19): мягкий лимит памяти — GC начинает работать активнее, чтобы удержаться под ним (спасает от OOM в контейнере).

Green Tea GC (по умолчанию с Go 1.26)

Тот же mark-sweep, но умнее сканирование памяти.

  • Было: GC обходил объекты «по графу» — по одному, прыгая по всей памяти вслед за указателями. Соседние объекты лежат далеко → постоянные кэш-промахи, процессор простаивает в ожидании данных из RAM (до 35% времени сканирования).
  • Стало: сканирование идёт локальными блоками — спанами по 8 КБ; объекты блока обрабатываются пачкой, данные лежат рядом → меньше кэш-промахов, лучше на многих ядрах (и векторные инструкции на новых CPU). Итог — на 10–40% меньше накладных расходов GC.

Как уменьшить нагрузку на GC

Меньше аллокаций в куче → реже и короче сборки: держать данные на стеке (не плодить лишние указатели), переиспользовать объекты (sync.Pool), меньше временных объектов в горячих циклах, заранее задавать ёмкость слайсов/мап.

Процесс, поток, системный вызов

TL;DR: процесс — запущенная программа со своей изолированной памятью; поток — линия исполнения внутри процесса (потоки одного процесса делят память); системный вызов (syscall) — просьба к ядру ОС сделать привилегированное (прочитать файл, открыть сокет).

Процесс — отдельная квартира (чужие не заходят). Потоки — жильцы одной квартиры (общая кухня = общая память). Системный вызов — звонок в ЖЭК, когда нужно то, что сам сделать не вправе.

Процесс vs поток

  • Память: процессы изолированы (не видят память друг друга); потоки одного процесса делят адресное пространство (куча, глобали, дескрипторы), но у каждого свой стек.
  • Общение: потоки — через общую память (нужна синхронизация: мьютексы и т.п.); процессы — через IPC: пайпы, сокеты, сигналы, разделяемая память.
  • Цена: создать процесс и переключаться между ними дороже (своё адресное пространство, сброс TLB/кэшей), чем между потоками. В ядре Linux и то, и другое — task; разница лишь в том, что́ они делят.

User space vs kernel space и syscall

Прикладной код работает в user space — ограниченный режим процессора: доступна только своя виртуальная память, часть инструкций запрещена. Ядро — в kernel space: доступ к железу, всей памяти, управлению процессами. Перейти из одного в другой можно только через системный вызов — приложение просит ядро сделать привилегированное. Это «дорого»: смена режима CPU + сохранение состояния. Разделение — ради безопасности: прикладной код не может сломать систему или залезть в чужую память.

Важные сисколлы: fork (создать процесс), read/write (I/O), epoll (асинхронно ждать готовности множества сокетов — основа сетевых серверов), mmap (отобразить файл/память).

Поэтому сисколлы и переключения потоков ОС дорого стоят, а планировщик Go держит переключения горутин в user space (без захода в ядро) и на тысячи соединений использует один epoll (netpoller / G-M-P).

Ресиверы методов: значение vs указатель

Короткий ответ: метод — это обычная функция, где ресивер передаётся первым аргументом по значению. Value receiver — копия значения: изменения снаружи не видны. Pointer receiver — копия указателя: можно менять то, на что он указывает, но не сам указатель у вызывающего.

Принцип №0, из которого выводится всё остальное. Вот такой метод и вызов:

func (c *Count) Increment() { ... }
count.Increment()

компилятор по сути превращает в обычную функцию:

func Increment(c *Count) { ... }
Increment(&count)

c внутри метода — локальная переменная. Присваивание в неё саму (c = &other) никогда не видно снаружи. Вопрос только в том, что именно скопировалось при вызове: значение целиком или указатель на него.

Value receiver — тебе выдали ксерокопию документа: черкай сколько хочешь, оригинал цел. Pointer receiver — тебе выдали записку с адресом, где лежит оригинал: по адресу можно прийти и исправить документ, но если ты перепишешь адрес в своей записке — у отправителя записка не изменится.

Сахар компилятора. x.Method() работает «в обе стороны»: если ресивер — указатель, а x — адресуемое значение, компилятор сам подставит (&x).Method(); если ресивер — значение, а x — указатель, подставит (*x).Method(). Из-за этого сахара кажется, что «всё само работает», пока не наткнёшься на неадресуемые значения или nil.

Разбор №1: c++ на указательном ресивере — почему не компилируется 
type Count int

func (c *Count) Increment() {
    c++ // invalid operation: c++ (non-numeric type *Count)
}

Внутри метода c имеет тип *Count — это указатель. c++ — попытка увеличить сам указатель, то есть арифметика указателей, которой в Go нет (в отличие от C). Рабочие варианты:

*c++      // парсится как (*c)++ — инкремент значения по адресу
(*c)++    // то же самое, но явно
*c += 1

*c++ в Go однозначен, потому что ++ — не выражение, а statement, применяющийся к операнду слева, то есть к *c.

Контрастный вариант — компилируется, но не работает:

func (c Count) Increment() {
    c++ // ок: c — число (Count), но это копия
}

var count Count
count.Increment()
fmt.Print(count) // 0 — оригинал не изменился

Почему (*c)++ меняет значение снаружи, хотя мы «разыменовали в обычное значение»? Разыменование само по себе ничего не копирует. *c — это не «достать значение наружу», а другое имя той же ячейки памяти, где живёт count. *c слева от = (или в ++) — «запиши по адресу», справа — «прочитай по адресу». Копия рождается только когда прочитанное значение куда-то кладут:

(*c)++    // изменилась память по адресу c → снаружи видно
v := *c   // ВОТ здесь копирование: значение легло в новую переменную v
v++       // увеличили копию → снаружи не видно
c = &x    // перезаписали локальную копию УКАЗАТЕЛЯ → снаружи не видно

Самопроверка: что выведет?

x := 10
p := &x
y := *p    // копия значения на этот момент
*p = 20
fmt.Println(x, y) // 20 10

y зафиксировал значение в момент присваивания и дальше живёт своей жизнью — связи с x он не хранит.

Разбор №2: nil-ресивер и почему init() ничего не инициализировал 
func main() {
    var v *a         // v == nil
    v.init()         // (1) НЕ паникует
    fmt.Println(v.b) // (2) panic: nil pointer dereference
}

type a struct{ b int }

func (v *a) init() {
    v = &a{b: 3}     // (3) перезаписали локальную копию — бесполезно
}

Три отдельные неочевидности:

(1) Вызов метода на nil-указателе легален. v.init() — это init(v), где v == nil. Передать nil в функцию — не преступление; паника случается не при вызове, а при разыменовании.

(3) v = &a{b: 3} перезаписывает локальную копию указателя. Указатель передался по значению; внутри метода v теперь смотрит на новую структуру, но в main он как был nil, так и остался.

(2) Поэтому v.b в main — разыменование nil → паника.

⚠️ «Починить» разыменованием тоже нельзя: *v = a{b: 3} внутри init сам паникует — мутировать через указатель можно только то, на что он уже указывает. Инициализировать nil-указатель вызывающей стороны из метода невозможно в принципе (понадобился бы **a). Идиоматично: конструктор func newA() *a { return &a{b: 3} } или метод, заполняющий поля уже существующего объекта (v := &a{}; v.init(), где init делает v.b = 3).

Легальность nil-ресивера используют и осознанно — как фичу:

func (l *List) Len() int {
    if l == nil { return 0 } // nil-список — это пустой список
    return l.n
}
Адресуемость: где сахар & не сработает (map, литералы, интерфейсы)

Автоматическое (&x).Method() возможно, только если у x можно взять адрес. Неадресуемы:

m := map[string]Count{"a": 0}
m["a"].Increment()   // ошибка компиляции: cannot call pointer method on m["a"]

Count(5).Increment() // литерал — ошибка
getCount().Increment() // результат функции — ошибка

Почему запрещено для map: элементы мапы физически переезжают в памяти при её росте, поэтому Go не даёт получить на них указатель. Обход — временная переменная (c := m["a"]; c.Increment(); m["a"] = c) или хранить в мапе указатели: map[string]*Count.

Method set: почему T не реализует интерфейс с pointer-методами 
type Incrementer interface{ Increment() }

var count Count
var i Incrementer = count  // ОШИБКА: Increment has pointer receiver
var i Incrementer = &count // ок

Правило: method set типа T — только методы с ресивером T; method set типа *T — методы с ресиверами и T, и *T.

Откуда асимметрия: имея *T, всегда можно разыменовать и получить T — поэтому *T умеет всё. А значение, лежащее внутри интерфейса, неадресуемо — компилятор не может автоматически взять &, поэтому T в интерфейсе pointer-методы не получает.

Ещё три ловушки: value receiver на nil, method value, копия мьютекса

1. Value receiver на nil-указателе — паника сразу на вызове:

func (v a) show() { fmt.Println("hi") } // поле даже не трогаем

var p *a
p.show() // PANIC: p.show() → (*p).show() — чтобы скопировать значение,
         // надо разыменовать nil ещё ДО входа в метод

Итого: nil + pointer receiver → вызов ок, паника только если внутри разыменуют; nil + value receiver → паника сразу.

2. Method value фиксирует ресивер в момент взятия:

type T struct{ n int }
func (t T) show() { fmt.Println(t.n) }

t := T{n: 1}
f := t.show // копия t зафиксирована ЗДЕСЬ
t.n = 2
f()         // 1, не 2 (с pointer receiver было бы 2)

Та же механика у defer t.show(): ресивер (как и аргументы) вычисляется в момент defer, а не в момент выполнения.

3. Копия ресивера копирует всё, включая мьютексы:

type Counter struct {
    mu sync.Mutex
    n  int
}

func (c Counter) Add() { // value receiver — БАГ
    c.mu.Lock()          // лочим КОПИЮ мьютекса
    c.n++                // мутируем копию
    c.mu.Unlock()
}

Гонка остаётся, счётчик не растёт. go vet ловит это как copylocks. Правило: тип содержит sync.Mutex, sync.WaitGroup и т.п. → только pointer receiver.

Строки: байты, руны, неизменяемость

Короткий ответ: строка в Go — неизменяемая последовательность байтов (обычно UTF-8) плюс маленький заголовок {указатель, длина}. len(s) — в байтах; s[i] — байт, а не символ; range — декодирует руны. Менять содержимое можно только через копию: []byte / []rune → правка → string(...).

Как устроена в памяти. Значение типа string — это структура из двух слов: указатель на массив байтов и длина. Поэтому копировать и передавать строки по значению дёшево (16 байт на 64-битной платформе), данные не дублируются. Сами байты неизменяемы — на этом держатся безопасный шаринг между горутинами без блокировок и использование строк как ключей map.

s := "го!"          // г(2 байта) + о(2 байта) + !(1 байт)
fmt.Println(len(s))  // 5 — БАЙТЫ, не символы
fmt.Println(s[0])    // 208 — первый байт 'г', тип byte (uint8)
s[0] = 'x'           // ошибка компиляции: строка неизменяема

for i, r := range s { // range декодирует UTF-8: i — БАЙТОВЫЙ индекс, r — руна
    fmt.Println(i, string(r)) // 0 г · 2 о · 4 !  — индексы скачут!
}

Руна — это rune = int32, номер символа в Unicode (code point). Подробнее про кодировку — в карточках Unicode и UTF-8. Посчитать символы: utf8.RuneCountInString(s) — O(n), но без аллокации; len([]rune(s)) — с аллокацией копии.

⚠️ Срез строки режет по байтам и спокойно разрубает многобайтовую руну пополам: "привет"[:3] → байты «п» + половина «р» → при печати «п�». Резать по символам можно только через []rune или аккуратно по границам из range.
Конкатенация: почему + в цикле — это O(n²) и причём тут Builder

Строка неизменяема ⇒ каждый s += x выделяет новую строку и копирует туда обе части. В цикле из n итераций суммарно копируется 1+2+…+n ≈ n²/2 байтов:

s := ""
for _, w := range words {
    s += w // на каждой итерации — новая аллокация + копирование всего s
}

Правильно — накапливать в изменяемом буфере и материализовать строку один раз:

var b strings.Builder
for _, w := range words {
    b.WriteString(w)
}
s := b.String() // амортизированно O(n)

// или, если это просто склейка слайса:
s := strings.Join(words, "")

Несколько + в одном выражении (a + b + c) компилятор склеивает за один проход — страшен именно цикл.

string ↔ []byte: когда копия, а когда нет + подстрока, которая держит гигабайт

Конвертации копируют. []byte(s) и string(b) создают копию данных — иначе изменяемый слайс сломал бы гарантию неизменяемости строки:

s := "hello"
b := []byte(s) // копия
b[0] = 'H'
fmt.Println(s, string(b)) // hello Hello — оригинал цел

Компилятор убирает копию в паре безопасных паттернов: m[string(b)] (поиск в мапе), for i, r := range []byte(s), сравнения вида string(b) == s.

А вот срез строки копию НЕ делает — подстрока разделяет тот же массив байтов. Отсюда классическая утечка-удержание:

data, _ := os.ReadFile("huge.log") // 1 ГБ
token := string(data[:10])          // здесь копия — ок
head  := bigString[:10]             // а здесь head держит ССЫЛКУ
                                    // на весь буфер bigString —
                                    // GC не освободит гигабайт,
                                    // пока жив head

Лечится явной копией: head := strings.Clone(bigString[:10]) (Go 1.18+).

Сравнение строк, регистр и коварный Unicode

== и < сравнивают байты лексикографически — быстро и без учёта языка. Без учёта регистра — strings.EqualFold(a, b) (правильнее, чем ToLower с обеих сторон: fold знает юникодные пары вроде K/k за пределами ASCII).

Нормализация: «é» может быть одной руной (U+00E9) или двумя (e + комбинируемый акцент) — выглядят одинаково, == даёт false, длины разные. Если на вход приходит пользовательский юникод и важно сравнение «как видит человек» — нормализуй (golang.org/x/text/unicode/norm, форма NFC).

Мелочь напоследок: нулевое значение строки — "", nil-строки не бывает; проверка на пустоту — len(s) == 0 или s == "" (эквивалентно).

Конкурентность

Конкурентность vs параллельность

TL;DR: конкурентность — умение управлять многими задачами сразу (переключаясь между ними); параллельность — физически выполнять их одновременно на нескольких ядрах.

Конкурентность возможна и на одном ядре (быстрое переключение создаёт иллюзию одновременности). Параллельность требует нескольких ядер. Конкурентность — про структуру программы, параллельность — про исполнение.

Один повар, жонглирующий тремя блюдами (конкурентность) vs три повара, каждый готовит своё (параллельность). Цитата Роба Пайка: «Concurrency is not parallelism».

Планировщик Go (G-M-P, work-stealing)

Короткий ответ: планировщик раскладывает тысячи горутин по горстке потоков ОС (модель M:N). Три сущности: G — горутина, M — поток ОС, P — «право на исполнение» с очередью горутин. Чтобы M выполнял Go-код, ему нужен P; число P = GOMAXPROCS (обычно = числу ядер).

P — стойки регистрации в аэропорту (их ровно GOMAXPROCS), G — пассажиры, M — сотрудники. Сотрудник обслуживает, только заняв стойку; если очередь у его стойки кончилась — забирает половину очереди у соседа.

Как это работает — G-M-P, очереди, сисколлы, вытеснение (со схемой)

Три сущности

G (goroutine) — сама горутина: её стек, состояние, что выполнять. M (machine) — поток операционной системы, который реально исполняет код. P (processor) — логический процессор планировщика: очередь готовых горутин + право исполнять. Правило: M крутит Go-код, только держа P → сколько P, столько горутин исполняется одновременно.

Очереди и приоритет выбора

У каждого P своя локальная очередь (LRQ, до 256 горутин, FIFO + оптимизация: только что разбуженную горутину кладут в отдельный «next»-слот и запускают вне очереди — данные ещё горячие в кэше). Плюс общая глобальная очередь (GRQ). Порядок, по которому P ищет следующую G: 1) раз в 61 тик заглянуть в GRQ (чтобы её горутины не голодали); 2) своя LRQ; 3) украсть половину LRQ у случайного P (work-stealing); 4) GRQ; 5) netpoller. Ничего нет — поток паркуется (park).

Блокирующий сисколл и сеть

Сисколл: когда горутина уходит в блокирующий системный вызов, её поток M «залипает» в ядре. Рантайм отцепляет P от этого M и отдаёт другому (свободному/новому) M, чтобы остальные горутины этого P не ждали (это handoff). Вернулся сисколл — бывший M пытается снова взять P; не вышло — горутина в очередь, лишний M засыпает.

Сеть — через netpoller: ожидание данных из сокета не блокирует поток. Горутина «паркуется», сокет регистрируется в netpoller (epoll/kqueue/IOCP), поток свободен и крутит другие горутины. Пришли данные — netpoller будит горутину. Отсюда фокус Go: пишешь простой синхронный код, а под капотом — эффективный асинхронный I/O; один сервер держит десятки тысяч соединений на горстке потоков.

Вытеснение (с Go 1.14)

Раньше планировщик был кооперативным: «жадная» горутина с тугим циклом без вызовов функций могла надолго захватить ядро. В 1.14 добавили асинхронное вытеснение: фоновый поток-сторож sysmon через сигнал ОС (SIGURG) прерывает горутину, крутящуюся дольше ~10 мс. sysmon также снимает P с застрявших в сисколлах и будит netpoller.

Состояния горутины

_Grunnable (готова, в очереди) → _Grunning (исполняется на M) → _Gwaiting (ждёт канал/мьютекс/таймер/сеть — вне очереди) либо _Gsyscall (в сисколле, блокирует M) → назад в _Grunnable; в конце _Gdead (структура g переиспользуется из пула).

Спрашивают на собесе: «что такое G, M, P?»; «сколько потоков создаст Go?» (≈ GOMAXPROCS + по одному на каждый заблокированный сисколл); «что делает рантайм, когда горутина блокируется на сисколле / ждёт сеть?»; «как Go вытесняет тугой цикл без вызовов функций?» (async preemption через SIGURG, с 1.14); «зачем work-stealing и почему крадёт половину?»; «что такое GOMAXPROCS?».

Каналы (channels)

Короткий ответ: канал — потокобезопасная «труба» для передачи данных между горутинами (одна кладёт ch <- v, другая забирает <-ch). Девиз Go: «не делите память — общайтесь через каналы». Небуферизированный — рандеву (отправитель ждёт получателя); буферизированный — очередь на N (не блокирует, пока есть место).

Небуферизированный — передать эстафету из рук в руки (встретились — передал). Буферизированный — положить на полочку на N мест, откуда заберут позже.

Буфер/небуфер, устройство (hchan), close, состояния, select

Небуферизированный vs буферизированный

Вся разница — в размере буфера: сколько значений канал может держать без читателя.

Небуферизированный make(chan T) — ёмкость 0, хранить негде. Это рандеву: значение идёт напрямую из рук отправителя в руки получателя, а ch <- v завершается ровно в тот момент, когда кто-то делает <-ch. Нет получателя — отправитель стоит сколько угодно. Отсюда важное: небуф. канал — это не «канал на 1 элемент», а на ноль. Годится как сигнал синхронизации.

Буферизированный make(chan T, N) — ячейка на N: отправитель кладёт значение и сразу уходит, получатель заберёт позже. Блокировка только если буфер полон при записи (тогда ведёт себя как небуферизированный) или пуст при чтении. Развязывает отправителя и получателя по скорости.

unbuf := make(chan int)     // ёмкость 0 — рандеву, ждём встречи
buf   := make(chan int, 1)  // ёмкость 1 — есть ячейка
buf <- 42        // не блокирует — легло в ячейку
v, ok := <-buf   // v=42, ok=true
close(buf)
v, ok = <-buf    // v=0, ok=false  (закрыт и пуст)

Зачем в генераторе нужна горутина

Генератор возвращает канал, из которого потом читают. Канал небуферизированный → запись ждёт читателя. Без горутины — замкнутый круг:

func calc(v int) <-chan int {
    ch := make(chan int)
    ch <- v * v   // ← навсегда застряли: ждём читателя
    close(ch)
    return ch     // ← сюда никогда не дойдём
}

Запись ждёт читателя → читатель появится только в main, но лишь после return → а return ждёт завершения записи. Go это ловит: fatal error: all goroutines are asleep - deadlock!. Горутина разрывает круг: запись уходит в фон и блокируется там, а calc немедленно возвращает канал.

С буфером можно и без горутины — место есть, запись не блокирует:

func calc(v int) <-chan int {
    ch := make(chan int, 1) // буфер на 1
    ch <- v * v             // не блокирует — легло в ячейку
    close(ch)
    return ch
}
⚠️ Но это не то же самое. С буфером v*v считается синхронно, прямо в момент вызова — вся конкурентность исчезает. Представь, что вместо умножения там HTTP-запрос на 2 сек: с горутиной три calc запускаются мгновенно, запросы идут параллельно → ~2 сек; без горутины main ждёт каждый по очереди → 6 сек. Горутина здесь и есть вся суть — делает calc неблокирующим «запусти и вернись», а канал служит способом забрать результат позже.

Устройство под капотом (hchan)

Канал — указатель на структуру hchan в куче. Внутри: кольцевой буфер, индексы записи/чтения, две очереди ждущих горутин (sudog), флаг закрытия и мьютекс на всю структуру — отсюда потокобезопасность (любая операция берёт lock).

type hchan struct {
    qcount   uint           // сколько сейчас в буфере
    dataqsiz uint           // размер буфера (0 у небуферизированного)
    buf      unsafe.Pointer // кольцевой буфер
    sendx    uint           // индекс записи в кольце
    recvx    uint           // индекс чтения в кольце
    recvq    waitq          // очередь горутин, ждущих чтения (sudog)
    sendq    waitq          // очередь горутин, ждущих записи (sudog)
    closed   uint32         // флаг закрытия
    lock     mutex          // мьютекс на всю структуру
}

Как идёт передача: при отправке горутина проверяет — закрыт ли канал, есть ли ждущий получатель в recvq, есть ли место в буфере. Небуф.: если есть ждущий получатель — пишет значение прямо в его стек и будит его (единственное место в рантайме, где одна горутина пишет в стек другой). Буф.: если есть место — кладёт в buf; полон и никто не ждёт — встаёт в sendq и блокируется.

Закрытие канала

close(ch) под локом: ставит closed=1, всем ждущим читателям отдаёт нулевое значение, всем ждущим писателям — панику, и будит их. Правила: закрывает только отправитель и только когда он единственный/последний писатель; закрывать со стороны читателя или при нескольких писателях — нельзя. Проверить, что закрыт, можно только чтением:

v, ok := <-ch   // ok == false → закрыт и пуст (v = нулевое)
for v := range ch {
    // цикл сам завершится, когда канал закроют и опустеет
}

Мнемоника: закрытый канал можно только читать; nil-канал только блокирует (а close(nil) — паника). «Чтение из закрытого не блокирует» — основа for range ch и приёма «в select ветку с закрытым каналом гасят через ch = nil».

⚠️ Когда все горутины засыпают (deadlock). Если все горутины программы заблокированы на каналах и никто не может продвинуться, рантайм это детектит и падает с fatal error: all goroutines are asleep - deadlock!. Это не паника — recover не спасёт.
func main() {
    ch := make(chan int)
    ch <- 1 // некому прочитать → блок навсегда
}
// fatal error: all goroutines are asleep - deadlock!
Здесь main одна и блокируется на отправке в небуферизированный канал без получателя. То же будет при чтении из пустого канала без отправителя, при wg.Wait() без Done() и т.п. Тонкость: ловится, только когда спят все — если хоть одна горутина жива (например, тикает таймер), утечка одной заблокированной горутины пройдёт незаметно.

select

select ждёт сразу несколько канальных операций. Готовые ветки перемешиваются и выбирается случайная; если готовых нет и есть default — выполняется он (неблокирующий режим), иначе горутина паркуется до готовности любой ветки. nil-канал в ветке делает её вечно не готовой (удобно «выключать» ветки).

Спрашивают на собесе: «буферизированный vs небуферизированный — в чём разница и когда что?»; «что внутри канала (hchan)?»; «что при чтении/записи/close на nil и закрытом канале?»; «кто и когда должен закрывать канал?» (только отправитель, один раз); «как узнать, что канал закрыт?» (v, ok := <-ch); «как работает select и зачем ch = nil?».

Гонка данных (race condition)

TL;DR: баг, когда несколько потоков/горутин одновременно обращаются к одним данным и хотя бы один пишет — результат становится непредсказуемым.

Пример: две горутины делают counter++. Эта операция не атомарна (прочитать → прибавить → записать); они могут «затереть» друг друга, и вместо 2 получится 1. Ищут детектором go test -race / go run -race (работает в рантайме, не на компиляции). Лечат синхронизацией: мьютекс, atomic, каналы.

Взаимоблокировка (deadlock)

TL;DR: ситуация, когда потоки навечно ждут друг друга и никто не может продолжить.

Классика: поток A захватил ресурс 1 и ждёт ресурс 2; поток B захватил ресурс 2 и ждёт ресурс 1. Оба замерли навсегда. В Go бывает и на каналах (все горутины уснули — рантайм печатает «all goroutines are asleep - deadlock!»).

Как избегать: захватывать блокировки всегда в одном порядке; использовать таймауты; дробить длинные транзакции; меньше ручных блокировок.

В БД (Postgres): дедлоки детектятся автоматически — СУБД находит цикл ожидания блокировок и откатывает одну транзакцию-«жертву» с ошибкой deadlock detected (по таймауту deadlock_timeout, ~1с). Приложение должно поймать ошибку и повторить транзакцию. Профилактика та же: блокировать строки всегда в одном порядке.

Два человека в узком коридоре уступают друг другу «после вас» — и стоят вечно.

Мьютекс, RWMutex, atomic и семафор

Короткий ответ: Mutex — «замок», пускает к данным только одного за раз. RWMutex — много читателей ИЛИ один писатель (быстрее, когда чтений ≫ записей). atomic — lock-free операции над одним словом (счётчики/флаги, дешевле мьютекса). Семафор — пускает до N одновременно.

Mutex — ключ от туалета (один зашёл — остальные в очереди); RWMutex — читальный зал (читают многие, но на уборку всех выгоняют); семафор — парковка на N мест.

Чем отличаются и когда что брать — с примерами

Mutex

mu.Lock() перед работой с общими данными, mu.Unlock() после (обычно defer mu.Unlock()). Пока один держит замок — остальные ждут.

var mu sync.Mutex
mu.Lock()
defer mu.Unlock()
balance += 10
  • Не реентерабельный: повторный Lock() в той же горутине → deadlock.
  • Нельзя копировать после использования (go vet ловит): копия — отдельный замок, синхронизации не будет. Передавать по указателю, не копировать структуру с мьютексом.
  • Взял — обязательно освободи по всем путям выхода (в т.ч. при ошибке/панике → defer).

RWMutex — когда чтений сильно больше

Два режима: RLock()/RUnlock() — много читателей одновременно; Lock()/Unlock() — писатель эксклюзивно (ждёт, пока уйдут все читатели, и не пускает новых). Выгоден, когда чтений ≫ записей и секция не крохотная. Минусы: тяжелее обычного Mutex (внутренний учёт читателей — точка конкуренции на многих ядрах), возможно «голодание» писателя. Для очень коротких секций обычный Mutex часто быстрее.

atomic — lock-free

Пакет sync/atomic — атомарные операции над одним словом без блокировки: Add, Load/Store, CompareAndSwap (CAS), типы atomic.Int64/Bool/Pointer, atomic.Value. Дешевле мьютекса, но защищает только одну переменную — не составную критическую секцию.

var n int64
atomic.AddInt64(&n, 1)      // потокобезопасный счётчик
v := atomic.LoadInt64(&n)

sync.Once и sync.Cond

sync.Once — гарантированно выполнить инициализацию один раз из нескольких горутин (once.Do(func(){ ... })). sync.Cond — условная переменная: горутины ждут (Wait) сигнала (Signal/Broadcast) о смене состояния; на практике часто заменяют каналами.

Семафор

Ограничивает число одновременных операций до N (напр. ≤ 10 запросов к внешнему API). В Go — буферизированный канал ёмкости N (занял слот перед работой, освободил после) или golang.org/x/sync/semaphore.

sem := make(chan struct{}, 10)
sem <- struct{}{}          // занять слот (ждёт, если 10 заняты)
go func() {
    defer func() { <-sem }() // освободить
    do()
}()

Атомарные операции (atomic)

TL;DR: операция «неделимая» — выполняется целиком за один шаг, её нельзя прервать на середине. Позволяет безопасно менять число из многих горутин без блокировок.

Пакет sync/atomic даёт атомарные Add, Load, Store, CompareAndSwap (CAS). Они реализованы на уровне инструкций процессора, поэтому быстрее мьютекса — но годятся только для простых вещей (счётчики, флаги). Мьютекс внутри сам построен поверх atomic (CAS).

Atomic — турникет, который либо пропустил и посчитал, либо нет; «полупройти» невозможно.

Паттерны конкурентности (fan-in / fan-out / pipeline)

Короткий ответ: типовые приёмы, как строить конкурентные программы на каналах. Генератор — функция возвращает канал с данными. Fan-out — несколько горутин читают из одного канала (распараллелили). Fan-in (merge) — несколько каналов сливаются в один. Pipeline — цепочка стадий через каналы. Везде главный вопрос: кто пишет и кто закрывает канал.

Fan-out — несколько кассиров на одну очередь. Fan-in — несколько очередей сходятся к одному выходу. Pipeline — конвейер: деталь идёт со станка на станок.

Генератор, fan-out, fan-in, pipeline — со схемами и разбором примера

Генератор (source)

Функция создаёт канал, запускает горутину, которая шлёт значения и закрывает канал, и сразу возвращает канал (только на чтение). Вызывающий читает через range.

func calc(v int) <-chan int {
    ch := make(chan int)
    go func() {
        ch <- v * v
        close(ch)   // закрыл — range у читателя завершится
    }()
    return ch
}

Fan-out — распараллелить чтение

Несколько горутин читают из одного канала for job := range jobs — задачи делятся между ними сами. Так масштабируют «тяжёлую» стадию.

// N воркеров читают из ОДНОГО канала jobs и делят задачи между собой
func fanOut(jobs <-chan int, workers int) <-chan int {
    results := make(chan int)
    var wg sync.WaitGroup
    wg.Add(workers)
    for i := 0; i < workers; i++ {
        go func() {                    // запускаем ровно `workers` горутин
            defer wg.Done()
            for job := range jobs {    // все читают из общего jobs
                results <- heavy(job)  // кто свободен — тот и берёт задачу
            }
        }()
    }
    go func() { wg.Wait(); close(results) }() // закрыть, когда jobs иссяк
    return results
}

Одна очередь jobs, N одинаковых воркеров разбирают её конкурентно — балансировка происходит сама собой (кто освободился, тот берёт следующий job). Число воркеров = и есть лимит параллелизма (это и есть worker pool). Закрывает results отдельная горутина после wg.Wait() — как и в fan-in.

Fan-in (merge) — слить каналы в один

Обратное: много входных каналов → один выходной. На каждый вход — своя горутина, которая читает его и форвардит в общий out. Ключевой момент: кто закроет out? Отдельная горутина ждёт завершения всех через WaitGroup и только тогда закрывает out — иначе for range out у читателя зависнет навсегда (закроем рано — паника «send on closed»).

func fanIn(chs ...<-chan int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup
    wg.Add(len(chs))
    for _, ch := range chs {
        go func(c <-chan int) {   // на каждый вход — своя горутина
            defer wg.Done()
            for el := range c {   // читаем, пока вход не закроют
                out <- el         // и форвардим в общий out
            }
        }(ch)
    }
    go func() {
        wg.Wait()   // дождались всех входов
        close(out)  // и закрыли out (иначе range в main зависнет)
    }()
    return out
}

Pipeline — конвейер стадий

Стадии соединены каналами: каждая читает вход, обрабатывает, пишет в свой выход и закрывает его. Сигнал завершения течёт по цепочке. Плюс легко вставить fan-out внутрь стадии.

// generator: числа → канал
func gen(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)          // закрыли выход, когда всё отдали
        for _, n := range nums {
            out <- n
        }
    }()
    return out
}

// стадия: читает вход, пишет квадраты в свой выход
func sq(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {       // тянем, пока вход не закроют
            out <- n * n
        }
    }()
    return out
}

func main() {
    // соединяем стадии вложением: gen → sq → sq
    for v := range sq(sq(gen(2, 3))) {
        fmt.Println(v)            // 16, 81  (2→4→16, 3→9→81)
    }
}

Каждая стадия — функция <-chan → <-chan: читает вход, обрабатывает, пишет в свой выход и defer close(out) при завершении. Стадии стыкуются просто подстановкой канала (sq(sq(gen(...)))). Сигнал завершения течёт по цепочке: gen закрыл выход → sq дочитал и закрыл свой → main вышел из range. Внутрь любой стадии легко вставить fan-out, если она «тяжёлая».

Спрашивают на собесе: «что такое генератор/fan-in/fan-out/pipeline?»; «кто закрывает канал при merge и зачем отдельная горутина с wg.Wait(); close(out)?»; «сохраняется ли порядок при fan-in?» (нет); «как ограничить параллелизм в fan-out?» (worker pool/семафор); «как прокинуть отмену через pipeline?» (ctx/done-канал в каждый select).

Сети

TCP и UDP

TL;DR: TCP — надёжная доставка с установкой соединения (как телефонный звонок); UDP — быстрая доставка без гарантий (как открытка).

TCP: устанавливает соединение (three-way handshake), гарантирует доставку и порядок (потерянные пакеты переотправляются), контролирует перегрузку. Медленнее из-за подтверждений. Для: HTTP, почта, БД, файлы.

UDP: просто шлёт датаграммы, ничего не гарантирует (могут потеряться/переставиться/задублироваться), зато без задержек на подтверждения. Для: видео/аудио-стриминг, игры, DNS, VoIP.

TCP — заказное письмо с уведомлением о вручении; UDP — крикнуть в толпу: быстро, но вдруг не услышат.

HTTP и версии

TL;DR: HTTP — текстовый протокол «запрос-ответ» поверх TCP для веба. Клиент шлёт запрос, сервер — ответ.

Запрос: стартовая строка (метод + путь + версия), заголовки, тело. Ответ: код состояния (200/404/500), заголовки, тело. Методы: GET (получить), POST (создать), PUT (заменить), PATCH (изменить), DELETE. Коды: 2xx успех, 3xx редирект, 4xx ошибка клиента, 5xx ошибка сервера.

Версии: HTTP/1.1 — одно соединение = один запрос за раз (keep-alive переиспользует соединение). HTTP/2 — мультиплексирование (много запросов по одному соединению параллельно), бинарный, сжатие заголовков. HTTP/3 — поверх UDP (QUIC), ещё быстрее.

HTTPS, SSL/TLS

TL;DR: TLS — шифрование поверх TCP, превращает HTTP в HTTPS. SSL — старое название TLS.

При установке соединения TLS-«рукопожатие» использует асимметричное шифрование (пара ключей: публичным шифруют, приватным расшифровывают) — чтобы безопасно договориться об общем сессионном ключе. Дальше данные шифруются быстрым симметричным ключом. Сертификат подтверждает, что сервер — тот, за кого себя выдаёт. HTTP — порт 80, HTTPS — 443.

Асимметрия — почтовый ящик: бросить письмо (публичный ключ) может любой, достать (приватный) — только владелец. Так передают секретный ключ, которым потом шифруют быстро.

DNS и NAT

TL;DR: DNS — «телефонная книга интернета», переводит имя google.com в IP-адрес. NAT — подмена приватных адресов на один публичный, чтобы много устройств выходили в интернет через один IP.

DNS: браузер спрашивает у DNS-серверов IP по домену (рекурсивно, с кэшированием). Типы записей: A (IPv4), AAAA (IPv6), CNAME (алиас), MX (почта), SRV (сервисы).

NAT: дома у всех устройств приватные адреса (192.168.x.x); роутер подменяет их своим публичным при выходе наружу и помнит, кому вернуть ответ. Экономит дефицитные IPv4 и скрывает внутреннюю сеть.

Базы данных

Индекс в БД

Короткий ответ: индекс — вспомогательная структура (по умолчанию B-tree), ускоряющая поиск строк, как алфавитный указатель в книге. Ускоряет чтение, но замедляет запись и занимает место. Без индекса — перебор всей таблицы (seq scan, O(n)); с индексом — O(log n). Полезен только на селективных условиях (в ответе мало строк).

Алфавитный указатель в конце книги: по слову сразу находишь нужные страницы, не листая всю книгу.

Виды индексов, лучшие/худшие случаи, boolean, когда НЕ используется

Как устроен — на примере

Индекс хранит значения колонки отсортированно вместе с указателем (TID) на строку. Поиск WHERE age = 31: бинарный поиск по индексу → TID → сразу нужная строка. Без индекса — seq scan всех строк.

-- было: Seq Scan (читает всю таблицу)
EXPLAIN ANALYZE SELECT * FROM users WHERE age = 31;

CREATE INDEX idx_users_age ON users (age);

-- стало: Index Scan (прыжок к нужным строкам)
EXPLAIN ANALYZE SELECT * FROM users WHERE age = 31;

Как БД выбирает способ поиска

  • Index Scan — идём по индексу и по одному достаём строки. Хорош при высокой селективности (мало строк).
  • Bitmap Index Scan — собираем все подходящие TID в битовую карту, затем читаем страницы по разу; умеет комбинировать несколько индексов (BitmapAnd/Or). Для средних выборок.
  • Seq Scan — если условие неселективно (возвращает большую часть таблицы), последовательное чтение быстрее индекса (меньше случайных чтений). Оптимизатор оценивает это по статистике.

Смотреть план — EXPLAIN ANALYZE: какой скан выбран, оценка vs факт строк, время.

Разновидности по построению

  • Составной (a, b) — по левому префиксу: помогает по a и a+b, но не по одному b. Порядок важен: сначала колонки по равенству, потом по диапазону.
  • Покрывающий (covering) — в индексе есть все нужные колонки → Index Only Scan без похода в таблицу (эффективнее после VACUUM).
  • Частичный WHERE условие — индексируем только часть строк; маленький и быстрый.
  • Функциональный (по выражению) — индекс не по колонке, а по результату функции/выражения: CREATE INDEX ON users (lower(email)). Нужен, когда в условии колонка обёрнута функцией: WHERE lower(email) = 'a@b.com' возьмёт такой индекс, а обычный по email — нет. Условие в запросе должно точно совпадать с выражением индекса. Частые кейсы: регистронезависимый поиск, date(created_at), вычисляемые поля.

Есть ли смысл индексировать boolean (true/false)?

Обычно нет: у булевой колонки очень низкая селективность (2 значения) — на «половину таблицы» оптимизатор всё равно выберет seq scan, обычный индекс не поможет и почти не будет использоваться. Смысл появляется, только если распределение сильно перекошено и ищут редкое значение — тогда берут частичный индекс на него: напр. is_deleted=true у 1% строк → CREATE INDEX ... WHERE is_deleted (крохотный индекс на нужные строки; на частое false индекс не нужен).

Когда индекс НЕ используется

  • Функция/приведение над колонкой: WHERE lower(x)=..., x::text=... → нужен функциональный индекс.
  • Низкая селективность — условие возвращает большую часть таблицы → seq scan дешевле.
  • Не левый префикс составного индекса (условие только по второй колонке).
  • LIKE '%abc' с ведущим % — B-tree бессилен (нужен GIN/триграммы pg_trgm).
  • Устаревшая статистика (нужен ANALYZE); очень маленькая таблица (seq scan и так мгновенный).

Кластерный vs некластерный

Кластерный индекс задаёт физический порядок строк на диске (один на таблицу). В MySQL/InnoDB первичный ключ — кластерный: строки лежат в B-tree по PK, вторичные индексы хранят PK. В PostgreSQL кластерных индексов нет — строки лежат в куче (heap) в произвольном порядке, все индексы вторичные (некластерные); команда CLUSTER лишь разово пересортирует таблицу (порядок дальше не поддерживается).

Спрашивают на собесе: «зачем индекс и чем платим?»; «почему индекс может не сработать?»; «виды индексов и когда что?» (B-tree/Hash/GiST/GIN/BRIN); «есть ли смысл индексировать boolean?» (только частичным на редкое значение); «покрывающий/частичный/составной/функциональный и левый префикс?»; «кластерный vs некластерный (MySQL vs Postgres)?»; «как понять, что запрос не использует индекс?» (EXPLAIN ANALYZE).

Транзакции и ACID

TL;DR: транзакция — группа операций «всё или ничего». ACID — 4 гарантии, которые делают её надёжной.
  • A (Atomicity, атомарность): либо все операции применились, либо ни одна (откат при ошибке).
  • C (Consistency, согласованность): транзакция переводит БД из одного корректного состояния в другое (не нарушая правил/ограничений).
  • I (Isolation, изоляция): параллельные транзакции не мешают друг другу (как будто идут по очереди).
  • D (Durability, надёжность): после подтверждения (commit) данные не потеряются даже при сбое.

Перевод денег: либо и списание, и зачисление, либо ничего — нельзя, чтобы деньги «исчезли» между счетами.

Как каждая буква реализована (Postgres) — глубина для сеньора
  • A — атомарность: через MVCC — версии строк помечены id транзакции; при откате транзакция просто помечается отменённой, и её версии строк становятся невидимыми для всех. Ничего физически «отматывать» не надо.
  • C — согласованность: constraints (PK/FK/UNIQUE/CHECK) + инварианты приложения. Тонкость, которую любят на собесе: C — единственная буква, которую БД не может обеспечить сама — «правильность» бизнес-инвариантов (баланс не отрицательный, долги сходятся) определяет приложение, БД лишь даёт инструменты.
  • I — изоляция: MVCC (снимки данных) + уровни изоляции — от Read Committed до Serializable (см. карточку «Уровни изоляции»).
  • D — надёжность: WAL (write-ahead log): изменения сначала пишутся в последовательный лог с fsync при commit, потом — в страницы данных. После сбоя БД доигрывает WAL от последнего checkpoint. На WAL же построены репликация и point-in-time recovery.
Спрашивают на собесе: «расшифруй ACID с примерами»; «как БД гарантирует durability?» (WAL + fsync + checkpoint); «как реализована атомарность?» (версии строк, отметка об отмене); «какая буква — не про БД?» (C — инварианты задаёт приложение).

Уровни изоляции и аномалии

TL;DR: чем строже изоляция параллельных транзакций, тем меньше «аномалий», но тем медленнее. Нужен баланс.

Аномалии (что может пойти не так при параллельности):

  • Потерянное обновление (lost update): две транзакции прочитали одно значение, обновили и закоммитили — одна затёрла изменения другой.
  • Грязное чтение (dirty read): прочитали чужие незакоммиченные изменения, которые потом откатились.
  • Неповторяемое чтение (non-repeatable read): перечитали ту же строку — значение уже другое (кто-то изменил и закоммитил).
  • Фантомное чтение (phantom read): перечитали по условию — строк стало больше/меньше (кто-то вставил/удалил).
  • Аномалия сериализации (serialization anomaly): итог группы параллельных транзакций не совпадает ни с одним последовательным порядком их выполнения.

Уровни (от слабого к строгому): Read Uncommitted → Read Committed (дефолт в Postgres) → Repeatable Read → Serializable (полная изоляция, как будто транзакции строго по очереди).

Потерянное обновление: на Read Committed его надо предотвращать самому — атомарным UPDATE ... SET x = x - 1 WHERE ... или SELECT ... FOR UPDATE; с Repeatable Read и выше Postgres сам ловит конфликт и падает с ошибкой сериализации (транзакцию нужно повторить).

Оптимистичные vs пессимистичные блокировки: пессимистичная — сразу лочим строку (SELECT FOR UPDATE); оптимистичная — не лочим, но при commit проверяем, не изменил ли кто версию (версионное поле), и если да — повторяем.

MVCC — многоверсионность

TL;DR: Postgres не заставляет читателей и писателей ждать друг друга — каждая транзакция видит свой «снимок» данных. Это и есть MVCC.
  • Суть: при UPDATE строка не переписывается на месте — создаётся новая версия, старая живёт, пока её видят другие транзакции. У версии есть метки xmin/xmax (кто создал/удалил).
  • Зачем: читатели не блокируют писателей и наоборот — SELECT видит версию, актуальную на момент его снимка. Изоляция без тяжёлых локов на чтение.
  • Snapshot isolation: на Repeatable Read транзакция работает с одним снимком на всю свою длину — отсюда в Postgres нет неповторяемых и фантомных чтений.
  • Цена: «мёртвые» версии (dead tuples) копятся → нужен VACUUM: убирает мёртвые версии, обновляет карту видимости (для Index Only Scan) и статистику. Autovacuum делает это автоматически в фоне по порогам (доля изменённых строк). Забьёшь — bloat (распухание таблиц и индексов) и замедление; при частых массовых UPDATE/DELETE его настраивают агрессивнее.

Как git: при изменении не стираешь старое, а добавляешь новую версию; каждый видит нужный «коммит».

Уровни изоляции — карточка «Уровни изоляции и аномалии».

Блокировки: оптимистичная и пессимистичная

Короткий ответ: два подхода к конкурентному изменению одних данных. Пессимистичная — сразу заблокировать строку (SELECT … FOR UPDATE): пока меняем — никто не тронет. Оптимистичная — не блокируем, но при записи проверяем, не изменил ли кто данные (по версии/timestamp), и если да — повторяем.

Пессимистичная — закрыть дверь на ключ, пока работаешь. Оптимистичная — работать при открытой двери, но перед сохранением проверить, не влез ли кто, и если влез — переделать.

Когда что, реализация, deadlock — и как решать

Пессимистичная

Блокируем ресурс заранее. В SQL — SELECT ... FOR UPDATE в транзакции: строка залочена до commit/rollback, другие с FOR UPDATE ждут. Надёжно при высокой конкуренции за одни строки (списание денег), но снижает параллелизм и рискует дедлоками/долгими ожиданиями. Варианты: FOR SHARE; SKIP LOCKED (взять свободные — для очередей на БД); NOWAIT (не ждать, сразу ошибка).

Оптимистичная

Не лочим. Держим колонку version (или updated_at). Читаем строку с версией, обновляем: UPDATE ... WHERE id=? AND version=?, увеличивая версию. Затронуто 0 строк → кто-то успел изменить (версия не совпала) → перечитываем и повторяем. Хорошо при редких конфликтах (много читают, редко пишут пересекающееся) — нет накладных на локи. В Postgres то же даёт Repeatable Read/Serializable (ловит конфликт и падает ошибкой сериализации).

-- пессимистичная
BEGIN;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
UPDATE accounts SET balance = balance - 10 WHERE id = 1;
COMMIT;

-- оптимистичная (version)
UPDATE accounts SET balance = balance - 10, version = version + 1
WHERE id = 1 AND version = 42;   -- 0 строк → повтор

Типы блокировок в Postgres (кто их берёт)

Табличные — берёт почти любая команда, различаются «тяжестью»: SELECT берёт лёгкий ACCESS SHARE, а DDL (ALTER/DROP, TRUNCATE, VACUUM FULL) — ACCESS EXCLUSIVE, который конфликтует со всеми. Поэтому ALTER под нагрузкой блокирует даже чтение — отсюда lock_timeout и CREATE INDEX CONCURRENTLY.

Строчные — берут UPDATE/DELETE и явный SELECT … FOR UPDATE (конфликтует сам с собой). Модификаторы: NOWAIT — не ждать, сразу ошибка; SKIP LOCKED — пропустить занятые строки (идиома очереди задач в БД); FOR SHARE — «читательская» блокировка строки.

Advisory (пользовательские) — не привязаны к данным: приложение само лочит произвольный числовой ключ через pg_advisory_lock(key) — удобно как распределённый лок между инстансами («только один воркер делает джобу»).

Важно: читатели и писатели не блокируют друг друга благодаря MVCC (читатель видит прошлую версию строки) — блокировки нужны писателям между собой и DDL.

Deadlock и как решать

Транзакция A залочила строку 1 и ждёт строку 2; B — наоборот → обе висят. Postgres детектит цикл и откатывает жертву с ошибкой deadlock detected (по deadlock_timeout, ~1с); приложение должно повторить транзакцию. Профилактика: брать блокировки всегда в одном порядке (напр. ORDER BY id); короткие транзакции; lock_timeout; меньше ручных локов.

Спрашивают на собесе: «оптимистичная vs пессимистичная — разница и когда что?»; «как реализовать каждую?» (SELECT FOR UPDATE / колонка version); «что при конфликте в оптимистичной?» (0 строк → повтор); «что такое deadlock и как БД его решает?» (детект + откат жертвы); «как избегать deadlock?» (единый порядок блокировок, короткие транзакции); «какие типы блокировок в PG и кто их берёт?» (табличные ACCESS SHARE/EXCLUSIVE, строчные FOR UPDATE/SKIP LOCKED, advisory).

Нормализация и денормализация

Короткий ответ: нормализация — раскладывать данные без дублирования (по таблицам со связями, «одна правда в одном месте»); денормализация — намеренно продублировать/агрегировать ради скорости чтения. Нормализация упрощает запись и целостность ценой JOIN'ов; денормализация ускоряет чтение ценой сложной поддержки согласованности.

Нормализация — хранить адрес клиента в одном месте и ссылаться на него; денормализация — вписать адрес в каждый заказ, чтобы не джойнить.

Нормальные формы 1NF→3NF (и BCNF), когда денормализовать

Нормальные формы

  • 1NF: все значения атомарны (никаких «списков в одной ячейке»), есть уникальный ключ. Нарушение: колонка phones = '111, 222'.
  • 2NF: 1NF + каждый неключевой атрибут зависит от всего составного ключа, а не от его части. Нарушение: в (order_id, product_id, product_name) — product_name зависит только от product_id → вынести товары в отдельную таблицу.
  • 3NF: 2NF + нет транзитивных зависимостей (неключевой зависит от другого неключевого). Нарушение: (id, city, zip), где zip определяет city → вынести справочник.
  • BCNF — усиленная 3NF (каждая детерминанта — потенциальный ключ). На практике целятся в 3NF.

Денормализация — когда осознанно нарушаем

Для высоконагруженного чтения и отчётов: дублируем/агрегируем данные, чтобы читать из одной таблицы без JOIN (счётчики, витрины, кэш-колонки, materialized view). Плата — при записи держать копии согласованными (триггеры/приложение/фоновые пересчёты), риск рассогласования. Правило: сначала нормализуй, денормализуй точечно под конкретный узкий запрос, когда упёрся в производительность.

Спрашивают на собесе: «что такое 1NF/2NF/3NF на примере?»; «зачем нормализовать?» (нет дублей и рассогласований); «когда денормализовать и чем платишь?»; «как поддерживать денормализованные данные согласованными?».

Репликация и шардирование

Короткий ответ: репликация — копии ОДНИХ И ТЕХ ЖЕ данных на разных серверах (надёжность + масштаб чтения); шардирование — РАЗНЫЕ части данных на разных серверах (масштаб объёма и записи). Часто вместе: каждый шард ещё и реплицируется.

Репликация — несколько одинаковых копий книги в разных читальных залах. Шардирование — тома энциклопедии (А–Й, К–П, Р–Я) по разным залам.

Мастер/реплики (когда, sync/async, лаг) и стратегии шардирования — со схемами

Репликация: мастер и реплики

Все записи идут на мастер (leader/primary); он передаёт поток изменений (WAL) на реплики (followers), которые обслуживают чтение.

Когда применять:

  • Масштаб чтения (read-heavy): много SELECT, мало записи → раскидываем чтение по репликам (напр. RDS read replicas).
  • Отказоустойчивость (HA): мастер упал → реплику промоутят в мастер (failover).
  • Гео-близость: реплики в разных регионах → чтение с ближайшей (меньше латентность).
  • Разгрузка: тяжёлую аналитику/бэкапы гоняют по реплике, не трогая мастер.

Синхронная vs асинхронная (главный трейдофф):

  • Асинхронная — мастер отвечает клиенту сразу, реплики догоняют потом. Быстро и устойчиво к падению реплик, но есть лаг репликации: реплика отстаёт → можно прочитать устаревшее (и при failover потерять последние коммиты). Дефолт для соцсетей.
  • Синхронная — мастер ждёт подтверждения реплики перед ответом. Нет потери при failover, но медленнее запись и уязвима к тормозам/падению реплики. Для банков.
  • Полусинхронная — ждём хотя бы одну реплику: баланс (e-commerce, бронирования).

Лаг репликации — частая боль: «read-your-writes» — записал на мастер, тут же прочитал с реплики и не увидел свою запись. Лечат: читать критичное с мастера, «липкие» сессии, ждать нужного LSN.

Шардирование (горизонтальное разбиение)

Когда нужно: данные или нагрузка записи не влезают в один сервер (реплики масштабируют только чтение). Тогда таблицу режут на шарды по ключу шардирования, каждый — на своём сервере.

Как обычно шардируют (стратегии):

  • По диапазону (range): id 1–1M → шард 1, 1M–2M → шард 2. Удобны range-запросы и сортировка, но легко получить горячий шард (все новые записи — в последний).
  • По хешу (hash): shard = hash(key) % N. Ровное распределение, нет горячих точек, но range-запросы невозможны, а при добавлении узла (меняется N) — массовое перераспределение.
  • Консистентное хеширование: шардов сильно больше, чем узлов, + таблица «шард→узел»; при добавлении узла переезжает лишь часть шардов, а не всё. Так делают Cassandra, DynamoDB, распределённые кэши.
  • Через справочник (directory): отдельная таблица «ключ→шард». Максимум гибкости (двигать/изолировать конкретных пользователей), но справочник — единая точка отказа и лишний lookup.
  • Гео/по тенанту: по региону или клиенту (data residency, изоляция крупных клиентов).

Выбор ключа критичен: он должен давать ровное распределение и попадать в частые запросы (иначе запрос «размажется» по всем шардам). Обычно шардируют по user_id/tenant_id, чтобы данные одного пользователя жили в одном шарде.

Минусы: запросы/JOIN между шардами дороги; распределённые транзакции сложны (2PC/Saga); ресардинг болезненный; горячие шарды при плохом ключе. Шардируют только когда реально упёрлись — сначала вертикальное масштабирование, индексы, кэш, реплики.

Горизонтальное vs вертикальное

Горизонтальное — делим строки (это и есть шардинг). Вертикальное — делим колонки/таблицы по назначению (редкие большие поля или подсистемы — в отдельную БД/сервис).

Функции, процедуры, триггеры, представления

Короткий ответ: код на стороне БД. Функция — возвращает значение, вызывается в запросе (SELECT f(x)). Хранимая процедура — выполняет действия и может управлять транзакцией (CALL p()), значение не возвращает. Триггер — функция, срабатывающая автоматически на INSERT/UPDATE/DELETE. View — сохранённый запрос как «виртуальная таблица».
Чем отличаются, зачем и подводные камни

Функция vs процедура

  • Функция — возвращает результат (скаляр, строку, таблицу), вызывается внутри SQL: SELECT total(order_id). Обычно выполняется в контексте вызвавшего запроса, транзакцией не управляет.
  • Процедура (Postgres 11+) — не возвращает значение, вызывается CALL; может делать COMMIT/ROLLBACK внутри — удобно для батчей/ETL.
  • Плюсы кода в БД: меньше round-trip'ов сеть↔БД, логика рядом с данными. Минусы: логика «размазана» между приложением и БД, сложнее версионировать/тестировать, вендор-лок (PL/pgSQL).

Триггеры

Функция, привязанная к таблице и событию: BEFORE/AFTER INSERT/UPDATE/DELETE, на строку (FOR EACH ROW) или на запрос. Применяют для аудита, автозаполнения (updated_at), проверки инвариантов, каскадов. Осторожно: скрытая логика (трудно отлаживать — «почему поменялось?»), просадка на массовых операциях, риск рекурсии.

Представления (VIEW)

  • VIEW — именованный сохранённый SELECT; обращаешься как к таблице, но данные считаются на лету при каждом запросе. Упрощает сложные запросы, скрывает детали/права. Данные не хранит, не ускоряет.
  • MATERIALIZED VIEW — хранит результат на диске (быстрое чтение тяжёлого запроса), но данные «застывают» — нужно REFRESH MATERIALIZED VIEW (вручную/по расписанию). По сути — форма денормализации/кэша.
Спрашивают на собесе: «функция vs процедура — разница?» (возвращает значение и вызывается в SELECT / CALL и управляет транзакцией); «что такое триггер и его минусы?» (скрытая логика, отладка, производительность); «VIEW vs MATERIALIZED VIEW?» (на лету vs хранит + REFRESH); «зачем/когда выносить логику в БД?».

Пул соединений к БД (database/sql)

Короткий ответ: sql.DB в Go — не соединение, а пул соединений (потокобезопасный, один на приложение). Соединение к Postgres — отдельный процесс на стороне БД, поэтому пул без лимитов под нагрузкой кладёт базу. Настраивают: SetMaxOpenConns, SetMaxIdleConns, SetConnMaxLifetime.

Каршеринг вместо личного авто на каждый запрос: взял машину из парка, съездил, вернул. Если парк не ограничить — парковка (БД) переполнится.

Настройки, типовые утечки, pgx и PgBouncer

Почему пул, а не соединение на запрос

Установка соединения — TCP + TLS + аутентификация (миллисекунды), а в Postgres ещё и форк процесса. Пул держит готовые соединения и переиспользует их. sql.Open не подключается к БД (проверка — db.Ping()), соединения создаются лениво.

Три ручки

  • SetMaxOpenConns(n) — потолок открытых соединений. Не задан → пул растёт неограниченно, под всплеском упрётся в max_connections Postgres (дефолт ~100) и начнёт валить ошибки всем.
  • SetMaxIdleConns(n) — сколько держать «тёплыми» в простое. Слишком мало → постоянные переподключения; обычно ставят равным или близким к MaxOpen.
  • SetConnMaxLifetime(d) — пересоздавать соединения старше d. Важно за балансировщиком/прокси (убивают idle-коннекты) и чтобы после failover трафик переполз на новый мастер.

Типовые утечки и зависания

  • Не закрыли rows: rows, _ := db.Query(...) без defer rows.Close() (или вышли из цикла до конца итерации) → соединение не вернулось в пул. Несколько таких — пул исчерпан, все запросы висят в ожидании свободного соединения.
  • Транзакция без Commit/Rollback — держит соединение (и блокировки в БД!) вечно. Паттерн: defer tx.Rollback() сразу после Begin (после Commit он безвреден).
  • Вложенный запрос в цикле по rows — на том же db берёт второе соединение на каждую итерацию: при MaxOpen=1 — самодедлок.

pgx и PgBouncer

pgx/pgxpool — родной драйвер Postgres: быстрее, бинарный протокол, свой пул. PgBouncer — внешний пулер между приложениями и БД: сотни клиентских коннектов мультиплексирует в десятки серверных (режим transaction pooling; ломает session-фичи вроде prepared statements на уровне сессии — надо учитывать).

Спрашивают на собесе: «что такое sql.DB — соединение?» (нет, пул); «какие лимиты настраивать и что будет без них?»; «почему запросы зависли, хотя БД жива?» (утечка rows/транзакций, пул исчерпан); «зачем ConnMaxLifetime?» (балансировщики, failover); «что такое PgBouncer и зачем?».

Партиционирование таблиц

Короткий ответ: разбиение одной большой таблицы на секции внутри одного сервера (в PG: PARTITION BY RANGE/LIST/HASH). Запросы читают только нужные секции (partition pruning), старые данные удаляются мгновенным DROP PARTITION. Не путать с шардированием — серверов не добавляет.

Один шкаф с документами, разложенными по папкам-месяцам: ищешь январь — открываешь одну папку; год истёк — выбрасываешь папку целиком, а не перебираешь листы.

Виды, зачем, грабли, отличие от шардирования

Виды (Postgres, декларативно с PG 10)

  • RANGE — по диапазону (типично — по времени: секция на месяц). Самый частый: логи, события, метрики.
  • LIST — по перечню значений (по региону, по тенанту).
  • HASH — равномерно по хешу ключа (когда нет естественного диапазона).

Что даёт

  • Partition pruning: WHERE created_at >= '2026-06-01' — планировщик читает только нужные секции, остальные даже не открывает.
  • Дешёвое удаление старого: DROP TABLE logs_2024_01 мгновенно вместо DELETE миллионов строк (который ещё и раздует таблицу мёртвыми кортежами — см. автовакуум).
  • Обслуживание по частям: VACUUM/индексация/бэкап маленькой секции вместо гигантской таблицы; индексы секций меньше и лучше влезают в кэш.

Грабли

  • Ключ партиционирования должен входить в PK/уникальные ограничения.
  • Запрос без фильтра по ключу сканирует все секции (медленнее, чем одна таблица).
  • Секции надо создавать заранее (автоматизируют: pg_partman, cron).

Партиционирование vs шардирование

Партиционирование — один сервер, цель — управляемость и скорость на больших таблицах. Шардирование — разные серверы, цель — масштаб записи/объёма за пределы одной машины. Часто вместе: внутри каждого шарда таблицы ещё и партиционированы.

Спрашивают на собесе: «чем партиционирование отличается от шардирования?»; «когда партиционировать?» (time-series, логи, ротация старого); «что такое partition pruning?»; «как удалить год старых логов, не положив прод?» (DROP PARTITION vs DELETE).

Инфраструктура и распределённые системы

Контейнеры vs виртуальные машины

TL;DR: VM — эмулирует железо и тащит целую гостевую ОС (тяжело, но сильная изоляция). Контейнер — процессы на общем ядре хоста, изолированные фичами ядра (лёгко, старт за секунды).

Контейнер — это не мини-ОС. Изоляция достигается двумя механизмами ядра Linux: namespaces (что процесс «видит» — свои PID, сеть, файловую систему) и cgroups (сколько ресурсов может брать — CPU, память). Общее ядро → почти нулевой оверхед, но изоляция слабее, чем у VM.

VM — отдельные дома с собственным фундаментом; контейнеры — квартиры в одном доме (общие стены/коммуникации = ядро), но со своими дверями и счётчиками.

Docker собирает контейнеры; multi-stage build даёт крошечные Go-образы.

Kubernetes — оркестрация контейнеров

TL;DR: система, которая запускает и следит за контейнерами на кластере машин: масштабирует, перезапускает упавшие, катит обновления без простоя.

Основные «кубики»:

  • Pod — минимальная единица: один+ контейнеров с общей сетью. Живёт недолго (умер — создался новый).
  • Deployment — держит N одинаковых подов, катит обновления (rolling update), откатывает.
  • Service — постоянный адрес к подам (поды меняются, service стабилен) + балансировка.

Probes (проверки здоровья): liveness — жив ли контейнер (нет → перезапуск); readiness — готов ли к трафику (нет → убрать из балансировки). Связка с graceful shutdown: под получает SIGTERM перед удалением — надо успеть доработать запросы.

Kubernetes — диспетчер такси: следит, чтобы нужное число машин было на линии, заменяет сломанные, распределяет заказы.

gRPC и protobuf

TL;DR: protobuf — компактный бинарный формат данных по схеме; gRPC — быстрый способ вызывать методы чужого сервиса поверх HTTP/2, используя protobuf.

Protobuf (Protocol Buffers): описываешь структуры в файле .proto, кодогенератор делает типы для твоего языка. Сериализация бинарная → в разы компактнее и быстрее JSON, схема строгая (версионируется).

gRPC — RPC-фреймворк от Google: по .proto генерятся клиент и сервер, общение идёт поверх HTTP/2 (мультиплексирование, стриминг). Поддерживает 4 типа вызовов: unary (запрос-ответ), server-stream, client-stream, bidirectional-stream. Стандарт для межсервисного общения; наружу для браузеров чаще REST/JSON (gRPC в браузере ограничен).

Сравнение с REST/RPC/SOAP — в банке, секция «Сети».

Kafka и брокеры сообщений

Короткий ответ: Kafka — распределённый реплицируемый лог коммитов: producer пишет в топик (разбит на партиции), consumer читает pull-моделью. Сообщения не удаляются после чтения (retention — дни/годы), их можно перечитывать. Масштаб — через партиции, отказоустойчивость — через репликацию и consumer-группы.

Не почтовый ящик (забрал письмо — его нет), а журнал: все записи лежат по порядку, каждый читатель держит свою «закладку» (offset) и читает в своём темпе.

Топики, партиции, оффсеты, группы, репликация — со схемой

Из чего состоит

  • Broker — сервер кластера; хранит партиции. Кластер = несколько брокеров.
  • Topic — именованный поток сообщений (логическая абстракция «куда писать/откуда читать»). Физически данные лежат в партициях.
  • Partition — часть топика; это и есть лог на диске. Партиции топика раскиданы по брокерам → горизонтальное масштабирование.
  • Offset — уникальный монотонно растущий номер записи внутри партиции (позиция). Имеет смысл только в своей партиции.
  • Producer пишет; Consumer читает; Consumer group — консьюмеры, делящие партиции топика между собой.
  • Replica — копия партиции на другом брокере (защита от потери данных).

Партиции и порядок

Порядок гарантирован только внутри партиции, не по всему топику. Сообщения с одинаковым ключом всегда попадают в одну партицию (хеш ключа) → порядок по ключу сохраняется. Параллелизм = число партиций: чтобы два консьюмера читали топик параллельно, нужно ≥2 партиций. Партиции можно добавлять на лету, но: это ломает порядок по ключу (меняется хеш→партиция) и удалить партицию нельзя (только весь топик).

Consumer groups

Внутри группы каждая партиция назначается ровно одному консьюмеру (чтобы не обрабатывать дважды). Если консьюмеров больше, чем партиций — лишние простаивают. Падение консьюмера → партиции автоматически перераспределяются между оставшимися (rebalance). Разные группы читают один топик независимо, каждая со своими оффсетами (сервис писем и сервис статистики читают одни события в своём темпе).

Ребаланс: при классическом (eager) протоколе на время переназначения партиций вся группа замирает (stop-the-world); cooperative переносит только затронутые партиции. Частые ребалансы (обработка дольше max.poll.interval.ms, флапающий под) → рост лага и дубли.

Оффсеты и коммиты

Консьюмер отмечает прогресс, коммитя offset (брокер хранит его в служебном топике). При рестарте продолжает с последнего закоммиченного. Важно: в момент времени только один закоммиченный offset на топик-партицию в группе — нельзя выборочно подтвердить отдельные сообщения (как ack в очередях). Ошибка обработки одного сообщения → его отправляют в отдельный топик на повтор (Dead Letter Queue — подробный разбор в карточке Паттерны надёжности), а offset двигают дальше.

Репликация и надёжность

Каждая партиция реплицируется replication factor раз. У партиции есть Leader (обслуживает продюсеров и консьюмеров) и Followers (реплицируют данные лидера, набор синхронных — ISR). Лидер упал → контроллер выбирает нового из ISR. Клиенты сначала спрашивают метаданные (кто лидер нужной партиции), подключаясь к любому брокеру.

Хранение

Партиция физически — это append-only лог, разбитый на сегменты (файлы). Есть активный сегмент для записи; дорос до лимита → новый. Старое удаляется по retention (времени/размеру), сегментами — это дёшево. Последовательная запись/чтение + zero-copy → очень высокая пропускная способность.

ZooKeeper / KRaft

Раньше метаданные кластера (живость брокеров, кто контроллер, лидеры партиций) хранил ZooKeeper — отдельная распределённая система; его падение почти = падение кластера. В новых версиях от ZooKeeper уходят в пользу встроенного KRaft (Kafka Raft) — метаданные внутри самой Kafka.

Kafka vs RabbitMQ

  • Kafka — лог: хранит сообщения (перечитывание, несколько независимых групп), pull-модель, огромный поток, event streaming.
  • RabbitMQ — классическая очередь: сообщение удаляется после ack, push-модель, гибкая маршрутизация (exchanges), удобна для «задача → воркеры».

Гарантии доставки

at-most-once (может потеряться), at-least-once (может задвоиться → обработчик должен быть идемпотентным), exactly-once (в Kafka — через идемпотентного продюсера + транзакции; дороже).

Спрашивают на собесе: «что такое топик/партиция/offset?»; «где гарантируется порядок?» (внутри партиции; ключ → одна партиция); «как масштабировать чтение?» (партиции + consumer group); «что при добавлении партиций / падении консьюмера?»; «как отметить обработку?» (commit offset; один на топик-партицию в группе; DLQ на ошибки); «зачем репликация и лидер?»; «Kafka vs RabbitMQ?»; «гарантии доставки?».

Redis и кэширование

TL;DR: Redis — сверхбыстрое in-memory хранилище «ключ-значение» (микросекунды): кэш, счётчики, сессии, rate limiter, распределённые локи. Быстрый, потому что всё в RAM и команды выполняет один поток (нет блокировок) на event loop. Плата: одна медленная команда блокирует всех.

Стол рядом с рабочим местом vs архив в подвале (БД): за частым — тянешься рукой, но на столе мало места, и надо решать, что убирать (eviction) и не устарело ли лежащее (инвалидация).

Структуры, персистентность, eviction, атомарность, локи, кластер, беды кэша

Структуры и типовые применения

  • string — кэш, счётчики (INCR атомарен); hash — объект по полям; list — очередь/лента; set — уникальные множества, пересечения; sorted set — рейтинги, «топ-N», rate limiter по времени; TTL на любой ключ — сессии, кэш.
  • Pub/Sub — рассылка «кто слушает сейчас» (без хранения); Streams — лог с consumer groups (мини-Kafka), когда сообщения терять нельзя.

Почему быстрый и в чём подвох однопоточности

Всё в RAM + один поток команд (нет локов и переключений контекста) + оптимизированные структуры. Подвох: KEYS *, SMEMBERS на миллион, тяжёлый Lua — блокируют все запросы. Правила: SCAN вместо KEYS, UNLINK вместо DEL для больших ключей. С Redis 6 I/O вынесен в потоки, но исполнение команд — по-прежнему одно.

Персистентность

RDB — периодические снапшоты (компактно, быстрый рестарт; теряешь данные с последнего снапшота). AOF — лог каждой операции (надёжнее, вплоть до fsync каждую секунду; файл больше, рестарт дольше). Часто вместе; для чистого кэша персистентность нередко выключают.

Память кончилась: eviction

По достижении maxmemory: noeviction (ошибка на запись, дефолт) / allkeys-lru / volatile-lru (только ключи с TTL) / allkeys-lfu / volatile-ttl / random. Для кэша — allkeys-lru/lfu; LRU приблизительный (сэмплирование). TTL ≠ eviction: протухание по времени и вытеснение по памяти — независимые механизмы.

Атомарность: pipeline vs MULTI/EXEC vs Lua

Pipeline — пачка команд без ожидания ответов: экономит round-trip'ы, не атомарен. MULTI/EXEC — команды подряд без вклинивания, но без отката и без чтения промежуточных значений (WATCH — оптимистичная проверка). Lua (EVAL) — атомарная процедура с логикой «прочитай-проверь-запиши»: rate limiter, безопасное снятие лока.

Распределённый лок

SET lock:job uuid NX EX 30 — атомарно «поставь, если нет, с TTL» (TTL обязателен — иначе упавший клиент держит лок вечно). Снимать Lua-скриптом «удали, только если value = мой uuid». Грабли: работа дольше TTL (нужно продление), failover реплики теряет лок; Redlock спорен — для критичных инвариантов лучше advisory lock в Postgres или etcd/ZooKeeper.

Масштабирование

Репликация master→replica (async, лаг) → чтение и резерв. Sentinel — наблюдатели с кворумом: автоматический failover (HA без шардинга). Cluster — шардинг: 16384 hash-слота по мастерам; multi-key операции — только в одном слоте (hash-теги {user:42}:cart).

Кэширование: паттерны и беды

  • Cache-aside (основной): читай кэш → мимо → БД → положи в кэш; при записи — обнови БД и удали ключ (не обновляй — меньше гонок). Write-through — писать в оба синхронно; write-behind — в кэш сразу, в БД асинхронно (риск потери).
  • Stampede — горячий ключ протух, сотни запросов пересчитывают одновременно → singleflight, блокировка, джиттер TTL.
  • Penetration — запросы к несуществующим ключам всегда летят в БД → кэшировать «пусто» с коротким TTL, Bloom-фильтр.
  • Avalanche — массовое протухание разом / падение кэша → джиттер TTL, прогрев, rate limit к БД.
Спрашивают на собесе: «почему Redis быстрый при одном потоке?» (RAM + event loop; но медленная команда блокирует всех); «что при переполнении памяти?» (maxmemory + eviction-политики); «pipeline vs транзакция vs Lua?»; «как сделать распределённый лок и какие грабли?» (SET NX EX + Lua-снятие, TTL, Redlock спорен); «Sentinel vs Cluster?» (failover vs шардинг); «stampede/penetration/avalanche — разница и защита?»; «Redis vs Memcached?» (структуры/персистентность/кластер vs простой многопоточный кэш).

LSM-tree vs B-tree

Короткий ответ: два способа хранить данные на диске. B-tree (Postgres, MySQL) пишет «на место» — read-optimized. LSM-tree (Cassandra, RocksDB, ClickHouse-подобные) копит записи в памяти и сбрасывает на диск неизменяемыми отсортированными файлами — write-optimized (запись последовательная).

B-tree — картотека: каждую карточку сразу ставишь на своё место (ходьба по ящикам). LSM — стопка входящих на столе: быстро кидаешь сверху, а по ночам сортируешь пачки и подшиваешь в архив.

memtable, SSTable, компакция, Bloom-фильтры, кто на чём

Как пишет LSM

  1. Запись → memtable (отсортированная структура в памяти) + WAL (на случай сбоя). Это быстро: RAM + последовательный лог.
  2. Memtable заполнился → сбрасывается на диск неизменяемым отсортированным файлом — SSTable.
  3. Фоновая компакция сливает SSTable между собой (как merge sort), выкидывая перезаписанные/удалённые версии.

Чем платим (чтение и компакция)

  • Ключ может лежать в memtable или любом из SSTable → чтение проверяет несколько файлов. Спасают Bloom-фильтры — вероятностная структура «в этом файле ключа точно нет / возможно есть»: пропускает большинство файлов без чтения.
  • Write amplification: компакция многократно переписывает данные — фоновая нагрузка на диск/CPU.
  • Удаление — не удаление, а запись «надгробия» (tombstone), физически чистится при компакции.

Кто на чём

B-tree: Postgres, MySQL/InnoDB — OLTP с балансом чтения/записи, диапазоны, транзакции. LSM: Cassandra/ScyllaDB, RocksDB/LevelDB (движок для Kafka Streams, CockroachDB, TiKV), HBase — write-heavy: события, метрики, time-series, очереди записи.

Спрашивают на собесе: «почему Cassandra быстро пишет?» (память + последовательный сброс SSTable); «что такое memtable/SSTable/компакция?»; «зачем Bloom-фильтр?» («точно нет» без чтения файла); «чем платим за быструю запись?» (чтение по нескольким файлам, write amplification); «B-tree vs LSM — когда что?» (read- vs write-optimized).

Документные БД (MongoDB)

Короткий ответ: хранят JSON-подобные документы (в Mongo — бинарный BSON) с гибкой схемой. Принцип моделирования: что читается вместе — хранится вместе (вложенность вместо JOIN). Атомарность — на уровне одного документа; репликация и шардинг из коробки.

Реляционка — картотека с перекрёстными ссылками (заказ в одном ящике, позиции в другом). Документная — папка клиента, где всё его лежит вместе: открыл одну папку — увидел всё.

Моделирование, replica set, шардинг, aggregation, транзакции, vs jsonb

Моделирование: вложить или сослаться

Вложить (embed) — когда данные читаются вместе и принадлежат родителю (позиции заказа, адреса пользователя): одно чтение, атомарное обновление. Сослаться (reference) — когда сущность разделяемая или растёт неограниченно (автор ↔ статьи; комментарии на миллион). Анти-паттерн — моделировать «как таблицы» и склеивать всё через $lookup: значит, нужна была реляционка.

Replica set и шардинг

  • Replica set: primary (запись) + secondaries; при падении primary — автоматические выборы нового. readPreference = чтение с secondary (eventual, лаг); write concern (w: majority) — сколько узлов подтвердили запись (скорость vs надёжность).
  • Шардинг: mongos-роутер → шарды (каждый — replica set); shard key выбирать как в SQL-шардинге: монотонный (время, ObjectId) → горячий чанк; hashed — ровно, но без range-запросов.

Aggregation pipeline и индексы

Конвейер стадий: $match → $group → $sort → $project → $lookup (left join). $match/$sort — в начало (используют индексы, режут объём). Индексы — B-tree: составные (левый префикс!), уникальные, TTL-индексы (авточистка), text/geo; explain(), COLLSCAN = seq scan.

Транзакции

Один документ (со всеми вложенностями) — атомарен всегда. Multi-document ACID — с 4.0/4.2, но дороже; правило: если транзакции нужны постоянно — модель данных, вероятно, реляционная.

Mongo vs Postgres jsonb

«Гибкая схема» есть и в Postgres: jsonb + GIN-индекс, рядом — обычные транзакции и JOIN. Mongo выбирают за модель целиком + масштабирование из коробки (replica set, шардинг, mongos). Честный ответ: «для гибких атрибутов внутри реляционного ядра — jsonb; для документной модели с горизонтальным масштабом — Mongo».

Спрашивают на собесе: «когда Mongo, когда реляционка?»; «embed vs reference?»; «что атомарно в Mongo?» (документ; multi-doc транзакции — дорогой запасной выход); «как устроен failover?» (replica set, выборы); «Mongo vs Postgres jsonb?»; «почему частый $lookup — плохой знак?».

CAP-теорема и консистентность

TL;DR: в распределённой системе при сбое сети (Partition) приходится выбирать между Consistency (все видят одни данные) и Availability (система отвечает). Все три сразу — нельзя.
  • C (Consistency) — любой читатель видит самую свежую запись.
  • A (Availability) — система всегда отвечает (пусть и не самыми свежими данными).
  • P (Partition tolerance) — работает, даже если часть узлов потеряла связь.

Сеть ненадёжна, поэтому P обязателен — реальный выбор между CP (жертвуем доступностью ради согласованности — банки) и AP (отвечаем всегда, данные «догонят» — соцсети). Eventual consistency («в конечном счёте согласованно») — компромисс AP-систем: реплики временно расходятся, но сходятся. Связано с синхронной (CP) vs асинхронной (AP) репликацией.

Синхронная vs асинхронная репликация — карточка «Шардирование и репликация».

Распределённые транзакции: 2PC и Saga

TL;DR: внутри одной БД транзакция — «всё или ничего». Между сервисами атомарности нет — нужны 2PC или Saga.

Проблема: заказ = списать оплату (сервис A) + зарезервировать товар (сервис B). Одной ACID-транзакцией это не обернуть — БД разные.

  • 2PC (two-phase commit): координатор спрашивает всех «готовы?» (prepare), затем «коммить» (commit). Даёт атомарность, но блокирующий и хрупкий: упал координатор между фазами — участники висят с локами. В микросервисах избегают.
  • Saga: цепочка локальных транзакций, у каждой — компенсирующее действие на откат. Оркестрация (центральный координатор ведёт шаги) или хореография (сервисы реагируют на события). Даёт итоговую согласованность, а не мгновенную.
  • Обязательно: идемпотентность шагов + Outbox для надёжной публикации событий (не потерять/не задвоить).

Поездка из рейсов разных авиакомпаний: не забронировался второй рейс — отменяешь (компенсируешь) первый.

Устойчивость: retry · circuit breaker · backpressure

TL;DR: как сервис переживает сбои и перегрузку зависимостей, не падая каскадом.
  • Retry с backoff: повтор при временной ошибке — с экспоненциальной задержкой и jitter, ограниченным числом попыток и только для идемпотентных операций.
  • Circuit breaker («предохранитель»): после серии ошибок внешнего сервиса перестаём его дёргать (состояние open → fail fast), через паузу пробуем (half-open). Защищает от каскадных падений и «добивания» лежащего сервиса.
  • Backpressure (обратное давление): потребитель не успевает — не копим бесконечно, а притормаживаем производителя: ограниченная очередь/буфер, блокирующая запись в канал, ответ 429. В Go естественно выражается небуферизированным/ограниченным каналом.
  • Timeout + context: любой внешний вызов — с таймаутом и отменой по ctx.

Предохранитель в щитке: при перегрузке размыкает цепь, чтобы не сгорел весь дом.

Паттерны надёжности: DLQ, Saga, Circuit Breaker, Inbox/Outbox

Короткий ответ: все четыре паттерна решают один класс проблем — как распределённая система остаётся согласованной и живой, когда нет общей ACID-транзакции и любой сетевой вызов может отвалиться или задвоиться. Outbox/Inbox — надёжно передать данные между БД и брокером. Saga — провести многошаговую бизнес-операцию без 2PC. DLQ — не дать одному битому сообщению встать поперёк очереди. Circuit breaker — не убить упавшего соседа лавиной запросов.

Все вместе — как конвейер на заводе с браком: Outbox/Inbox — датчики на входе и выходе каждого станка, гарантирующие «деталь либо прошла целиком, либо не прошла вообще, но не потерялась»; DLQ — отдельная тележка, куда откладывают бракованные детали, чтобы не тормозить весь конвейер; Saga — инструкция «если деталь испортили на пятом станке — откатить операции первых четырёх»; Circuit breaker — предохранитель, который останавливает подачу деталей на сломанный станок, пока его не починят.

Outbox: как не потерять событие между «записал в БД» и «отправил в брокер»

Проблема (dual write): обработчик должен и сохранить данные в БД, и опубликовать событие в брокер — это два разных хранилища, одной транзакцией их не покрыть. Любой порядок операций небезопасен:

1) сначала БД, потом брокер:  БД записалась → сервис упал до publish → событие потеряно навсегда
2) сначала брокер, потом БД:  событие ушло → сервис упал до commit → потребители узнали о том, чего не было

Решение — Transactional Outbox: вместо прямой публикации пишем событие в обычную таблицу outbox той же транзакцией, что и бизнес-данные — атомарность обеспечивает сама БД (это уже не dual write, а одна транзакция в одну БД). Отдельный процесс (relay) читает outbox и публикует в брокер.

BEGIN;
  INSERT INTO orders (id, status) VALUES (42, 'created');
  INSERT INTO outbox (id, topic, payload, sent) VALUES (gen_uuid(), 'orders', '{...}', false);
COMMIT;
-- отдельный процесс: SELECT ... WHERE sent=false → publish → UPDATE outbox SET sent=true

Как читает relay: либо поллинг (простой periodic SELECT по outbox, есть у себя в коде), либо CDC (change data capture — читает журнал транзакций БД, например Debezium поверх WAL/binlog — не нагружает БД лишними SELECT'ами, публикует почти в реальном времени).

⚠️ Outbox даёт только at-least-once. Если relay опубликовал событие, но упал до UPDATE ... sent=true — при рестарте отправит его повторно. Дублирование не устранено, а сдвинуто на сторону потребителя — отсюда необходимость Inbox/идемпотентности на другом конце.

Inbox: зеркало Outbox на стороне потребителя

Проблема: Outbox (и брокеры в целом) дают at-least-once доставку — потребитель обязан быть готов получить одно и то же событие дважды. Если обработка = запись в БД + побочный эффект (списать деньги, отправить письмо), «просто быть идемпотентным» на словах — легко забыть на практике.

Решение — Transactional Inbox: зеркало Outbox с другой стороны. У входящего сообщения есть уникальный message_id. Проверка «уже обработано?» и сама бизнес-запись делаются в одной транзакции:

BEGIN;
  INSERT INTO inbox (message_id) VALUES ($id)   -- ON CONFLICT DO NOTHING /
                                                  -- PK-конфликт → уже обработано, откат
  -- если вставка прошла (не дубль) — применяем бизнес-эффект:
  UPDATE accounts SET balance = balance - 100 WHERE id = $account;
COMMIT;

Если message_id уже есть в inbox — вставка конфликтует, транзакция откатывается без применения эффекта: сообщение молча дедуплицируется. Outbox на одном конце + Inbox на другом вместе дают эффективно exactly-once на уровне бизнес-эффекта, даже когда сам транспорт (брокер) гарантирует только at-least-once.

DLQ (Dead Letter Queue): что делать с сообщением, на котором обработчик стабильно падает

Poison message — сообщение, которое обработчик не может обработать в принципе (битый формат, неожиданные данные, баг в коде на конкретном payload). Без защиты — зацикливание: получил → упал → брокер редоставит → получил → упал… В Kafka это особенно больно: пока не продвинут offset партиции, застревает вся партиция (нельзя подтвердить сообщения выборочно, как ack в очередях).

Схема лечения — по нарастающей:

  1. Ретраи с backoff — сначала пробуем ещё раз (вдруг сбой временный: сеть моргнула, соседний сервис перегружен). В Kafka задержку нативно не сделать — эмулируют отдельными retry-топиками (orders-retry-5m, orders-retry-30m), сообщение перекладывается туда и возвращается позже.
  2. Ретраи исчерпаны (N попыток) → сообщение уходит в DLQ-топик/очередь вместе с метаданными ошибки (стектрейс, время, попытки) — а исходная партиция/очередь продолжает работать дальше.
  3. DLQ обязательно мониторится (алерт на рост глубины) и разбирается человеком: чинят баг → реплеят сообщения из DLQ обратно в основной топик.
⚠️ DLQ без мониторинга и процедуры реплея — просто кладбище потерянных данных. Завести очередь мало — нужен процесс: кто и как часто её разбирает.

Saga: распределённая транзакция без 2PC

Проблема: бизнес-операция задевает несколько сервисов/БД (списать деньги — сервис A, зарезервировать товар — сервис B) — общей ACID-транзакции нет физически (2PC технически возможен, но блокирующий и хрупкий: упал координатор между фазами — участники висят с локами; в микросервисах почти не используют).

Saga — цепочка локальных транзакций, у каждой заранее определено компенсирующее действие на откат. Пример — оформление заказа:

Шаги:        создать заказ → списать деньги → зарезервировать товар → создать доставку
Если резерв не удался — компенсации в ОБРАТНОМ порядке:
             вернуть деньги ← отменить заказ

Даёт не мгновенную, а итоговую (eventual) согласованность: какое-то время система видит промежуточное «полусостояние» (заказ создан, деньги ещё не списаны).

Два стиля организации — ключевой вопрос на собесе:

  • Оркестрация — есть центральный координатор (сервис-оркестратор или engine типа Temporal/Camunda), который явно вызывает каждый шаг и знает весь сценарий целиком. Плюс: весь процесс виден в одном месте, легко отлаживать и менять порядок шагов. Минус: лишний компонент, точка связности (оркестратор знает про все сервисы).
  • Хореография — координатора нет: каждый сервис публикует событие о своём завершении, следующий сервис в цепочке на него подписан и реагирует сам. Плюс: нет единой точки отказа/связности, сервисы независимы. Минус: сценарий «размазан» по коду N сервисов — сложно понять целиком и тяжело отлаживать («а кто вообще слушает это событие?»).

Практическое правило: простая цепочка из 2–3 шагов — хореография (не оверинжинирить). Длинный сценарий с ветвлениями, таймаутами, ручными шагами (модерация) — оркестрация, часто на специальном движке (Temporal, Camunda, AWS Step Functions), а не самописный код.

Обязательные спутники Saga: шаги должны быть идемпотентны (шаг может выполниться повторно после сбоя/ретрая) и события — публиковаться надёжно, отсюда прямая связь с Outbox выше.

Поездка из рейсов разных авиакомпаний: не забронировался второй рейс — отменяешь (компенсируешь) первый, а не звонишь в «единую транзакцию бронирования», которой не существует.

Circuit Breaker: не добивать лежачий сервис

Внешняя зависимость деградировала (тормозит или падает) — наивные ретраи усугубляют проблему: каждый клиент долбит её повторными запросами, добивая последним ресурсом (потоки/соединения), которого ей и так не хватает. Circuit breaker — конечный автомат из трёх состояний вокруг вызова:

CLOSED (норма) ──ошибок больше порога──▶ OPEN (fail fast, вызовы не идут)
   ▲                                          │
   │                                    прошёл таймаут
   │                                          ▼
   └──пробный запрос успешен──── HALF-OPEN (пропускаем 1 пробный запрос)
                                       │
                                пробный неудачен → назад в OPEN
  • Closed — обычная работа, вызовы идут напрямую, ведём счётчик ошибок.
  • Open — ошибок за окно больше порога → перестаём звать сервис вообще, сразу возвращаем ошибку (fail fast) — экономим время клиента и не долбим упавший сервис.
  • Half-open — после паузы пробуем один запрос: прошёл — возвращаемся в closed; нет — обратно в open (и снова пауза, обычно растущая).

Отличие от retry с backoff: retry — «попробовать снова эту же операцию»; circuit breaker — «перестать пробовать вообще, пока сервис не восстановится». Часто комбинируют: retry с backoff внутри breaker'а, но breaker сам решает, пропускать ли попытку.

Спрашивают на собесе: «зачем нужен Outbox, если можно просто отправить событие после коммита?» (dual write — либо потеряешь событие, либо закоммитишь то, чего не было); «Outbox даёт exactly-once?» (нет, только at-least-once — для эффекта на потребителе нужен Inbox); «что такое poison message и чем грозит в Kafka?» (партиция встаёт целиком, offset не двигается); «оркестрация или хореография для Saga — когда что?»; «чем circuit breaker отличается от retry?» (перестаём пробовать вообще, а не пробуем снова).

Как отвечать на собесе

Поведенческие / скрининговые вопросы (STAR)

Короткий ответ: на «расскажи самую сложную задачу», «конфликт в команде», «твоя ошибка» отвечай по схеме STAR: Ситуация → Задача → Действие (что именно ты сделал) → Результат в цифрах. И добавь «как решил бы сейчас» — показать рост.

Интервьюер проверяет не саму историю, а как ты думаешь: как диагностируешь, какие варианты взвешиваешь и почему выбираешь именно это.

Разбор на примере «самая сложная инженерная задача»

Формула STAR + инженерная глубина:

  • Ситуация / задача (1–2 фразы): система, метрика, в чём затык — «латентность p99 росла до 2с под пиком, теряли заказы».
  • Сложность: почему нетривиально — конкурентность, объём данных, легаси, неочевидный трейдофф. Именно это интервьюер и оценивает.
  • Действие: что именно ты сделал — как диагностировал (метрики → pprof → трейс), какие варианты рассмотрел и почему выбрал этот (показал, что взвешивал трейдоффы, а не гуглил наугад).
  • Результат: в цифрах — «p99 упал с 2с до 120мс, заказы перестали теряться».
  • «Как сейчас»: покажи рост — «вынес бы это в отдельный сервис / добавил бы circuit breaker с самого начала».

Красные флаги: «мы просто добавили серверов»; рассказ без своей роли («команда сделала»); без цифр; без объяснения почему выбрано именно это решение.
