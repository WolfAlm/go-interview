Сумма квадратов через канал (deadlock)

Что выведет этот код?

Код задачи
func main() {
    ch := make(chan int)
    wg := &sync.WaitGroup{}
    wg.Add(3)
    for i := 0; i < 3; i++ {
        go func(v int) {
            defer wg.Done()
            ch <- v * v
        }(i)
    }
    wg.Wait()
    var sum int
    for v := range ch {
        sum += v
    }
    fmt.Printf("result: %d\n", sum)
}

Вывод: fatal error: all goroutines are asleep - deadlock!

Почему: канал небуферизированный, читателя ещё нет → все 3 горутины виснут на ch <- v*v. defer wg.Done() сработает только после отправки, поэтому wg.Wait() не разблокируется. Плюс range по незакрытому каналу.

Способ 1 — буфер на все значения + close
func main() {
    ch := make(chan int, 3)      // ← буфер на 3 — отправка не блокирует
    wg := &sync.WaitGroup{}
    wg.Add(3)
    for i := 0; i < 3; i++ {
        go func(v int) {
            defer wg.Done()
            ch <- v * v
        }(i)
    }
    wg.Wait()
    close(ch)                    // ← закрыли, чтобы range завершился
    var sum int
    for v := range ch {
        sum += v
    }
    fmt.Printf("result: %d\n", sum) // 5
}

Буфер вмещает все 3 значения, поэтому горутины не ждут читателя; wg.Wait() дожидается их, а close позволяет range остановиться. Минус: надо заранее знать размер — не годится, если число значений неизвестно.

Способ 2 — без буфера: wg.Wait()+close в отдельной горутине
func main() {
    ch := make(chan int)
    wg := &sync.WaitGroup{}
    wg.Add(3)
    for i := 0; i < 3; i++ {
        go func(v int) {
            defer wg.Done()
            ch <- v * v
        }(i)
    }
    go func() {          // ← ждём и закрываем в фоне,
        wg.Wait()        // ←   пока main уже читает из канала
        close(ch)        // ←
    }()              // ←
    var sum int
    for v := range ch {
        sum += v
    }
    fmt.Printf("result: %d\n", sum) // 5
}

Главное: wg.Wait() нельзя звать в main до чтения — тогда некому забрать значения из небуф. канала, и всё висит. Выносим ожидание и close в отдельную горутину, а main сразу входит в range и читает параллельно отправкам.

✦ Идеал — генератор + fan-in
func calc(v int) <-chan int {
    ch := make(chan int)
    go func() { ch <- v * v; close(ch) }()
    return ch
}
func fanIn(chs ...<-chan int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup
    wg.Add(len(chs))
    for _, ch := range chs {
        go func(c <-chan int) {
            defer wg.Done()
            for el := range c {
                out <- el
            }
        }(ch)
    }
    go func() { wg.Wait(); close(out) }()  // закрывает, когда все источники иссякли
    return out
}
func main() {
    chs := make([]<-chan int, 0, 3)
    for i := 0; i < 3; i++ {
        chs = append(chs, calc(i))
    }
    var sum int
    for v := range fanIn(chs...) {
        sum += v
    }
    fmt.Printf("result: %d\n", sum) // 5
}

Каждый источник — свой генератор-канал; fanIn сливает их в один и сам закрывает out. Масштабируется на любое число источников и встраивается в pipeline. На собесе: достаточно способа 2 или 3; fan-in показывает знание идиом.

Способ 1 — sync.WaitGroup
func main() {
    a := 5000
    var wg sync.WaitGroup      // ←
    wg.Add(a)                  // ←
    for i := 0; i < a; i++ {
        go func(i int) {       // ← i аргументом (до Go 1.22 иначе общий i)

    wg.Wait()                  // ← дождались всех

Классика: WaitGroup считает запущенные горутины, wg.Wait() держит main, пока все не сделают Done. i передаём аргументом — до Go 1.22 замыкание захватило бы общий i, и почти все напечатали бы 5000.

Способ 2 — канал завершения
func main() {
    a := 5000
    done := make(chan struct{}, a)   // ← канал завершения
    for i := 0; i < a; i++ {
        go func(i int) {
            fmt.Println(i)
            done <- struct{}{}       // ← сигнал «готово»
        }(i)
    }
    for i := 0; i < a; i++ {         // ← ждём ровно a сигналов
        <-done
    }
}

Если нужен только факт «готово» — можно каналом: каждая горутина шлёт в done, main читает ровно a раз. WaitGroup тут по сути тот же счётчик.

✦ Идеал — errgroup
func main() {
    a := 5000
    var g errgroup.Group   // golang.org/x/sync/errgroup
    for i := 0; i < a; i++ {
        i := i
        g.Go(func() error {
            fmt.Println(i)
            return nil
        })
    }
    _ = g.Wait()
}

errgroup — тот же WaitGroup, но с возвратом ошибок и лимитом параллелизма (g.SetLimit). Суть задачи: main сам горутины не ждёт — нужен явный барьер (WaitGroup / канал / errgroup). Порядок вывода не гарантирован.

Отправка в небуф. канал до читателя

Как отработает эта программа?

Код задачи
func main() {
    ch := make(chan int)
    ch <- 1
    go func() {
        fmt.Println(<-ch)
    }()
}
🔍 Показать: что выведет / как отработает

Блокируется главная горутина на ch <- 1: канал небуферизированный, читателя ещё нет (горутина ниже даже не стартовала) → send ждёт рандеву → рантайм видит, что все спят → deadlock. Горутина-читатель не запускается.

Способ 1 — буфер на 1
func main() {
    ch := make(chan int, 1)   // ← буфер на 1 — send не блокирует
    ch <- 1
    go func() {
        fmt.Println(<-ch)
    }()
}

Буфер убирает deadlock: значение ложится в ячейку без читателя. Но без синхронизации main может завершиться раньше, чем горутина напечатает — если вывод важен, дождись её (способ 2).

Способ 2 — читатель раньше отправителя + ожидание
func main() {
    ch := make(chan int)
    done := make(chan struct{})   // ←
    go func() {
        fmt.Println(<-ch)   // читатель стартует первым
        close(done)         // ←
    }()
    ch <- 1                 // ← теперь есть кому принять
    <-done                  // ← дождались печати
}

Правильный порядок для небуф. канала: сначала запускаем читателя, потом отправляем — рандеву состоится. done гарантирует, что main не выйдет до печати.

✦ Идеал — генератор
func gen() <-chan int {
    ch := make(chan int, 1)
    ch <- 1
    return ch
}
func main() {
    fmt.Println(<-gen())   // читатель — вызывающий, порядок гарантирован
}

Функция возвращает канал, а читатель (main) синхронно забирает значение — чисто, без ручного done. Главное: send в небуф. канал без читателя = deadlock.

Буфер вмещает оба значения, поэтому ch <- true не блокируют, даже если читатель ещё не появился.

Отдельного «идеала» тут нет: число сигналов известно (2), поэтому читатель в отдельной горутине, принимающий ровно столько (способ 1) — это и есть идиома; небуф. канал не течёт памятью и не требует close. На собесе: назвать, что deadlock именно на первой отправке.

Гонка: общий a и запись в map

Что выведет этот код?

Код задачи
var globalMap = map[string][]int{"test": {}, "test2": {}, "test3": {}}
var a = 0

func main() {
    wg := sync.WaitGroup{}
    wg.Add(3)
    go func() {
        wg.Done()
        a = 10
        globalMap["test"] = append(globalMap["test"], a)
    }()
    go func() {
        wg.Done()
        a = 11
        globalMap["test2"] = append(globalMap["test2"], a)
    }()
    go func() {
        wg.Done()
        a = 12
        globalMap["test3"] = append(globalMap["test3"], a)
    }()
    wg.Wait()
    fmt.Printf("%v", globalMap)
    fmt.Printf("%d", a)
}
🔍 Показать: что выведет / как отработает

Что напечатает — недетерминировано.

(1) wg.Done() стоит в начале горутины, поэтому wg.Wait() разблокируется до записей в map. Из-за этого main может напечатать пустые срезы и a=0 (горутины не успели), а может — уже заполненные значения. Как повезёт с планировщиком.

(2) Три горутины пишут в общий map (и читают общую a) — это data race, поведение не определено. Рантайм может упасть с fatal error: concurrent map writes (или concurrent map read and map write из-за Printf), но это не гарантия: детектор best-effort — ловит гонку, только если две операции реально пересеклись в момент проверки внутреннего флага.

Например, два возможных вывода:

А — горутины не успели (Wait вернулся сразу):
map[test:[] test2:[] test3:[]]0

Б — успели отработать:
map[test:[10] test2:[11] test3:[12]]12

Из-за гонки по a значения в срезах могут «перемешаться» (напр. test:[12]), возможны и промежуточные варианты (часть ключей пустые), а изредка — fatal error. Ключи в %v Go печатает отсортированными.

Способ 1 — Mutex + Done в конце
func main() {
    wg := sync.WaitGroup{}
    var mu sync.Mutex          // ←
    wg.Add(3)
    run := func(key string, val int) {  // ← значение аргументом, без общей a
        defer wg.Done()        // ← Done в конце, ПОСЛЕ записи
        mu.Lock()              // ←
        globalMap[key] = append(globalMap[key], val)
        mu.Unlock()            // ←
    }
    go run("test", 10)
    go run("test2", 11)
    go run("test3", 12)
    wg.Wait()
    fmt.Printf("%v", globalMap)
}

Мьютекс защищает запись в map; defer wg.Done() — в конце, чтобы wg.Wait() дождался записей; значение передаём аргументом, а не через общую a.

Способ 2 — единственный писатель через канал
func main() {
    type kv struct { k string; v int }
    ch := make(chan kv, 3)          // ←
    var wg sync.WaitGroup
    wg.Add(3)
    write := func(k string, v int) {
        defer wg.Done()
        ch <- kv{k, v}              // ← пишем в канал, не в map
    }
    go write("test", 10)
    go write("test2", 11)
    go write("test3", 12)
    wg.Wait(); close(ch)
    for e := range ch {             // ← единственный писатель в map — main
        globalMap[e.k] = append(globalMap[e.k], e.v)
    }
    fmt.Printf("%v", globalMap)
}

Горутины шлют пары в канал, а в map пишет только main — гонок нет вовсе, мьютекс не нужен.

✦ Идеал — sync.Map
func main() {
    var globalMap sync.Map     // потокобезопасная мапа
    var wg sync.WaitGroup
    wg.Add(3)
    add := func(key string, val int) {
        defer wg.Done()                  // Done в конце
        globalMap.Store(key, []int{val}) // каждый пишет СВОЙ ключ
    }
    go add("test", 10)
    go add("test2", 11)
    go add("test3", 12)
    wg.Wait()
    globalMap.Range(func(k, v any) bool {
        fmt.Println(k, v)
        return true
    })
}

sync.Map потокобезопасна и хорошо ложится сюда: каждая горутина пишет свой ключ (непересекающиеся ключи) — как раз один из её штатных сценариев. Тонкости: значения — any, на чтении нужен type-assert; и sync.Map не делает атомарным append к срезу одного ключа из нескольких горутин (Load+append+Store не атомарны) — там уже нужен mutex. И два бага оригинала никуда не деваются: wg.Done() в начале и общая a (значение — только аргументом). Помни: даже запись по разным ключам в обычный map — гонка, потому и нужна sync.Map или mutex.

Реализуй MultiSearch: параллельно ищет query во всех SearchFunc и возвращает результат первого успешного (без ошибки) сразу, не дожидаясь остальных и отменив их. Если все вернули ошибку — вернуть последнюю.

func MultiSearch(ctx context.Context, query string, sfs []SearchFunc) (Result, error) {
    ctx, cancel := context.WithCancel(ctx)
    defer cancel()
    type res struct { r Result; err error }
    out := make(chan res, len(sfs)) // буфер = len → горутины не зависнут
    for _, sf := range sfs {
        go func(sf SearchFunc) {
            r, err := sf(ctx, query)
            out <- res{r, err}
        }(sf)
    }
    var lastErr error
    for range sfs {
        if r := <-out; r.err == nil {
            return r.r, nil
        } else {
            lastErr = r.err
        }
    }
    return Result{}, lastErr
}

Ключевое: (1) буфер out = len(sfs) — иначе после раннего return непрочитанные горутины зависнут (утечка); (2) WithCancel + defer cancel() — первый успех отменяет остальных; (3) читаем ровно len раз. Добивка: «а если канал небуферизированный?» → зависнут.

func (m *Monitor) CheckAll(ctx context.Context, devices []string) ([]DeviceStatus, error) {
    if len(devices) == 0 {
        return nil, fmt.Errorf("empty device list")
    }
    results := make([]DeviceStatus, len(devices)) // своя ячейка на устройство
    g, ctx := errgroup.WithContext(ctx)
    g.SetLimit(m.maxWorkers)                       // лимит параллелизма
    for i, name := range devices {
        i, name := i, name
        g.Go(func() error {
            st, err := m.checkOne(ctx, name)       // checkOne возвращает ошибку
            if err != nil {
                return err

            results[i] = st                        // без гонки и без мьютекса
            return nil
        })

    }
    return results, nil

// разворот односвязного списка — итеративно, O(n), O(1) памяти
func reverse(head *Node) *Node {
    var prev *Node
    for cur := head; cur != nil; {
        cur.next, prev, cur = prev, cur, cur.next
    }
    return prev // prev — новая голова
}

// N-й с конца — два указателя с разрывом N (один проход)
func nthFromEnd(head *Node, n int) *Node {
    fast := head
    for i := 0; i < n; i++ {
        if fast == nil {
            return nil // список короче N
        }
        fast = fast.next
    }
    slow := head
    for fast != nil {
        fast, slow = fast.next, slow.next
    }
    return slow
}

// пересечение двух слайсов — O(n+m) через map-счётчик
func intersect(a, b []int) []int {
    cnt := map[int]int{}
    for _, x := range a {
        cnt[x]++
    }
    var res []int
    for _, x := range b {
        if cnt[x] > 0 {
            res = append(res, x)
            cnt[x]--
        }
    }
    return res
}

Разворот: идём по списку, разворачивая указатели next назад; prev в конце — новая голова. Главное — не потерять cur.next до перезаписи (в Go выручает множественное присваивание).

N-й с конца: «быстрый» указатель уходит вперёд на N, затем двигаем оба до конца — «медленный» окажется на N-м с конца. Один проход, без подсчёта длины.

Пересечение: счётчик по первому слайсу, потом идём по второму и берём, пока счётчик > 0 (учитывает дубликаты). O(n+m) времени, O(n) памяти.

    v := reflect.ValueOf(i)
    switch v.Kind() {
    case reflect.Ptr, reflect.Map, reflect.Slice,
        reflect.Chan, reflect.Func, reflect.UnsafePointer:
        return v.IsNil()
    }
    return false // значимые типы (int, struct…) nil не бывают
}

Проверка Kind() обязательна: IsNil() на значимом типе паникует. И вторым тактом скажи: если бизнес-коду приходится так проверять — сломан дизайн на стороне, которая вернула типизированный nil; правильное лечение — return nil явно. Но сначала покажи, что механику знаешь.

Честная сериализация и ошибки возвращаются вызывающему — но каждый Log ждёт диск под локом, при всплеске нагрузки все горутины выстраиваются в очередь. Озвучь это и переходи к асинхронному варианту.

Идеал — очередь + единственный писатель (актор)
type SequentialLogger struct {
    wrapped Logger
    msgs    chan string
    done    chan struct{} // закроется, когда писатель дописал всё
    mu      sync.RWMutex  // защищает closed от гонки с Close
    closed  bool
}

func NewSequentialLogger(wrapped Logger) *SequentialLogger {
    sl := &SequentialLogger{
        wrapped: wrapped,
        msgs:    make(chan string, 1024), // буфер сглаживает всплески
        done:    make(chan struct{}),
    }
    go func() {                    // единственный писатель — сериализация «бесплатно»
        defer close(sl.done)
        for m := range sl.msgs {   // выйдет, когда канал закрыт И дочитан
            if err := sl.wrapped.Log(m); err != nil {
                fmt.Fprintf(os.Stderr, "log write: %v\n", err) // ошибки — см. разбор
            }
        }
    }()
    return sl
}

func (sl *SequentialLogger) Log(message string) error {
    sl.mu.RLock()                  // RLock: Log-и идут параллельно, Lock берёт только Close
    defer sl.mu.RUnlock()
    if sl.closed {
        return errors.New("logger closed")
    }
    sl.msgs <- message             // буфер полон → backpressure (блокируемся)
    return nil
}

func (sl *SequentialLogger) Close() error {
    sl.mu.Lock()
    if sl.closed {
        sl.mu.Unlock()
        return nil                 // идемпотентный Close
    }
    sl.closed = true
    close(sl.msgs)                 // безопасно: новых отправителей уже не будет
    sl.mu.Unlock()

    <-sl.done                      // дождаться, пока писатель дольёт очередь
    return sl.wrapped.Close()      // и только потом закрыть файл
}

Что решает обёртка (проговори первым): (1) гонку конкурентных Log из прошлой задачи — пишет одна горутина, строки не перемешиваются; (2) вызывающие не ждут диск — запись развязана очередью; (3) вся сложность (закрытие, переполнение) собрана в одном месте, а FileLogger остаётся тупым писателем. Это декоратор над Logger + паттерн «единственный писатель» — обёртка работает с любой реализацией интерфейса.

Почему флаг + RWMutex, а не «просто закрыть канал»: send в закрытый канал — паника. Канал нельзя закрывать, пока возможны отправители. Поэтому Log проверяет флаг и держит RLock на время отправки, а Close под полным Lock выставляет флаг и закрывает канал — в этот момент активных отправителей гарантированно нет (они бы держали RLock).

Ошибки записи: раз запись асинхронна, Log уже не может вернуть ошибку диска — он возвращает только «logger closed». Варианты, назови и выбери: молча терять (плохо), stderr + метрика (по умолчанию ок), канал ошибок/callback наружу, синхронный режим с ответным каналом (теряется весь смысл обёртки).

Переполнение очереди: блокироваться — ничего не теряем, но при деградации диска встанет всё приложение; дропать (через select с default) + счётчик потерянных — логер не роняет сервис. Для логов чаще выбирают drop, для аудита — блокировку. Главное — показать, что выбор осознанный.

Close без потерь: закрыть канал → дождаться done (писатель дочитал очередь) → закрыть вложенный логер. Если закрыть файл раньше — хвост очереди упадёт с ошибками. Опционально: таймаут на дренаж через select.

Бонусы для сеньор-уровня: батчинг (писатель забирает из канала пачку и пишет одним вызовом), bufio.Writer + периодический Flush, метрики глубины очереди.

(3) MIN(id) в подзапросе NULL не бывает, поэтому NOT IN здесь безопасен; в Postgres чаще пишут через ROW_NUMBER() OVER (PARTITION BY email ORDER BY id) и удаление rn > 1. Доп. вопрос: как не допустить дублей дальше? — UNIQUE-индекс (создавать CONCURRENTLY на проде).

Эталон / подсказка
-- НЕ одной транзакцией. Батчи по PK + keyset-пагинация:
last := 0
for {
    UPDATE t SET col = <new>
    WHERE id > last AND id <= last + 10000
      AND col IS DISTINCT FROM <new>   -- пропускаем уже правильные
    COMMIT                             -- фиксируем каждый батч
    last += 10000                      -- запоминаем прогресс (резюмируемость)
    sleep(50ms)                        -- троттлинг: даём репликам догнать
}

Золотое правило: резать на батчи по 1–10к строк и коммитить каждый. Один UPDATE на 10 млн строк = долгий лок, гигантский WAL, лаг реплик и риск отката всей работы.

Гонять по PK (keyset-пагинация): WHERE id > :last ORDER BY id LIMIT N — предсказуемо и по индексу PK; храни last, чтобы после сбоя продолжить с места.

Не трогать уже правильные строки: col IS DISTINCT FROM <new> — меньше мёртвых строк (bloat) и меньше WAL.

Троттлинг: пауза между батчами; следить за лагом реплик и autovacuum, тормозить если растут. После большого апдейта — VACUUM, при сильном bloat — pg_repack.

Тяжёлая трансформация: вместо UPDATE — новая колонка/таблица → backfill батчами → переключение (как в миграциях); онлайн-инструменты pg_repack (PG), pt-online-schema-change (MySQL).

Если правим только часть (например «сегодняшние» ряды) — коррелирующий индекс/условие резко сокращает объём перебора.
